### SettingsSection — internal layout

```
SettingsSection (VStack, spacing: 8)
  Text("Settings")                        — .headline, .padding(.horizontal), .padding(.top, 4)
  LaunchAtLogin.Toggle("Launch at Login") — .padding(.horizontal), .padding(.vertical, 4)
  Divider (thin, within section)
  Text("Modifier Hotkeys")               — .headline, .padding(.horizontal), .padding(.top, 4)
  HStack                                  — Plain Dictation picker row
    Text("Plain Dictation")  .body
    Spacer()
    Picker (menu style)      — options: Fn+Shift, Fn+Control, Fn+Option
  .padding(.horizontal).padding(.vertical, 4)
  HStack                                  — AI Cleanup picker row
    Text("AI Cleanup")       .body
    Spacer()
    Picker (menu style)      — options: Fn+Shift, Fn+Control, Fn+Option
  .padding(.horizontal).padding(.vertical, 4)
  Text("Fn-based hotkeys for keyboards without a dedicated Fn key are not supported.")
    — .caption, .secondary, .padding(.horizontal), multilineTextAlignment(.leading)
```

Layout rules:
- `LaunchAtLogin.Toggle("Launch at Login")` is the simplest usage — use single-argument form, do not wrap in custom HStack
- `Picker` style: `.pickerStyle(.menu)` — renders as a compact popup button, consistent with macOS menu bar app conventions
- Picker label on left, Picker control on right (Spacer between) — mirrors HotkeySettingsView row pattern exactly
- The external keyboard note is a `.caption` in `.secondary` color, left-aligned, multiline if needed
- No button or CTA needed in the settings section — all controls are direct toggles/pickers

Source: CONTEXT.md D-03/D-10/D-11; RESEARCH.md §Pattern 2 (LaunchAtLogin.Toggle); CLAUDE.md §Supporting Libraries (LaunchAtLogin-Modern); default applied for Picker style and note typography

## Interaction States

### Launch at Login Toggle

| State | Visual | Behavior |
|-------|--------|----------|
| Off (default) | `LaunchAtLogin.Toggle` unchecked | App does not launch at login |
| On (user opted in) | `LaunchAtLogin.Toggle` checked | macOS shows system notification "Dicticus" added to Login Items |
| State drift (user disabled via System Settings) | Toggle automatically reflects system state on next dropdown open | No app-side caching — always reads from SMAppService |

Rules:
- Off by default per D-02 — LaunchAtLogin-Modern initializes to disabled state
- Do NOT cache toggle state in UserDefaults — SMAppService is the source of truth
- macOS system notification on add is expected behavior — do not suppress or explain it in-app

Source: CONTEXT.md D-02/D-04; RESEARCH.md §Pattern 2, §Pitfall 5

### Modifier-Only Hotkey Picker

| State | Visual | Behavior |
|-------|--------|----------|
| Default | "Plain Dictation" picker shows "Fn+Shift", "AI Cleanup" picker shows "Fn+Control" | Matches Phase 3 defaults per D-09 |
| User changed | Picker shows selected combo | Selection persisted in UserDefaults or AppStorage |
| Both pickers same value | Allowed — no conflict detection required in Phase 5 | User responsibility to avoid conflicts |
| Selection applies immediately | No save/apply button — picker onChange triggers HotkeyManager update | Direct binding pattern |

Picker options (all three pickers — plain dictation and AI cleanup):
1. "Fn + Shift" (default for plain dictation)
2. "Fn + Control" (default for AI cleanup)
3. "Fn + Option"

Rules:
- Use `Picker("label", selection: $binding)` with `.pickerStyle(.menu)` — no inline or segmented style
- Picker label on left at `.body` weight — same as HotkeySettingsView row labels
- No "None" / disabled option — if user wants no modifier hotkey, they rely solely on KeyboardShortcuts combos (which remain active in parallel)

Source: CONTEXT.md D-09/D-10/D-11; RESEARCH.md §Pattern 1 (ModifierCombo presets); default applied for onChange binding pattern

### CGEventTap / Modifier Hotkey Activation

No user-visible UI change during modifier key detection. The CGEventTap runs in the background. Activation states route to the existing HotkeyManager push-to-talk state machine — menu bar icon transitions follow the Phase 3 contract (mic.fill red while held, waveform.circle pulsing while processing).

Source: CONTEXT.md D-08; Phase 3 UI-SPEC §Menu Bar Icon States (carried)

## DMG Background Image Specification

The styled DMG (D-07) requires a background image for the Finder window.

| Property | Value | Rationale |
|----------|-------|-----------|
| Canvas size | 1320 × 800 px | 2× of the 660 × 400pt window for Retina displays (RESEARCH.md Pitfall 4) |
| Format | PNG | Required by create-dmg |
| Background | Dark grey or near-black — `#1C1C1E` (macOS system groupedBackground in dark mode) | Matches native macOS dark DMG convention; neutral for any desktop wallpaper |
| App icon position | Left-center: 360 × 400 px (at 2× = 720,800 logical center) — icon at 256×256 pt, centered at (180pt, 200pt) | Follows create-dmg defaults |
| Applications symlink position | Right-center: centered at (480pt, 200pt) in logical coordinates | Standard macOS DMG convention |
| Arrow indicator | Subtle rightward arrow glyph between icon and Applications folder — `→` in SF Pro light weight at ~48pt, color `#FFFFFF` at 40% opacity | Indicates drag direction without aggressive visual styling |
| Text label below icon | "Drag Dicticus to Applications" — SF Pro Regular 14pt, color `#FFFFFF` at 70% opacity | Brief instruction; matches minimal tone |
| File path | `scripts/dmg-background.png` | Referenced in build-dmg.sh per RESEARCH.md §Pattern 3 |

Rules:
- No gradient backgrounds — flat color only
- No custom branding illustrations — app icon provides the only visual identity
- The image must be exactly 1320 × 800 px (2× the window dimensions declared in the build script)
- Export from any design tool as PNG, no transparency in background layer (fully opaque)

Source: CONTEXT.md D-07; RESEARCH.md §Pattern 3 (create-dmg options), §Pitfall 4 (Retina background size); Claude's discretion (color, arrow, label — all within "DMG background image design" discretion area)
