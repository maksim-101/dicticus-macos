## 1. Prompt-Leakage Findings (variants for D-04)

### Hard constraint: Gemma has no system role

Per [official Gemma prompt structure docs](https://ai.google.dev/gemma/docs/core/prompt-structure), Gemma 3/4 instruction-tuned models support **only `user` and `model` roles**. There is no separate system channel — all directives must live inside the single user turn. This rules out the "promote HELVETISMS to system block" mitigation pattern. Channel separation must come from explicit labeled sections and ordering inside the user turn.

`[CITED: ai.google.dev/gemma/docs/core/prompt-structure]`

### What is currently in the user turn (CleanupPrompt.build, lines 49–116)

The prompt is a single user turn assembled in this order:

1. `INSTRUCTION:` (the user's chosen polish instruction; default is preservation-light)
2. `DICTIONARY:` (optional)
3. `LANGUAGE:` (de/en)
4. `STYLE:` (Swiss orthography — when swiss+de, lines 73–75)
5. `HELVETISMS:` (the problematic block — lines 76–92)
6. `STRICT:` (currency anti-flip — when currency detected in `text`)
7. `INPUT:` (the actual transcription)
8. The model is then asked to continue from `OUTPUT:`

The `HELVETISMS:` block today (lines 83–92) contains THREE stacked signals:

- Preservation directive ("Preserve the speaker's dialect register exactly")
- 13-pair NEGATIVE trap list ("auf→uf, ausgeflogen→usgfloge, …")
- 30-word `SwissHelvetisms.words` POSITIVE reference list ("If — and only if — the speaker actually used a Swiss word, keep it as-is (reference list: …)")

The 2026-04-27 failure shows Gemma output Swiss forms that were NOT in the trap list (`hie`, `Siite`, `wahrschiinli`, `wuer`) and were NOT in the positive reference list — strong empirical signal that the model is treating the entire `HELVETISMS:` block as a *vocabulary-domain anchor* rather than a *do-not-translate* anchor. This is the prompt-leakage hypothesis the spike must validate.

`[ASSUMED]` Cause hypothesis. The spike is what proves it.

## 3. Helvetism-Delta Gate Implementation Details

### Function signature (mirror existing gate)

In `Shared/Services/CleanupService.swift`, in the same extension as `gateLLMOutput`:

```swift
extension CleanupService {

    /// Phase 20.08 D-06..D-08. Pre-Levenshtein dialect-suppression gate.
    /// Returns `llmOutput` if it introduces zero unsolicited Swiss dialect
    /// forms relative to `rulesCleaned`; otherwise returns `rulesCleaned`
    /// (LLM rejected as Swiss-ifier).
    ///
    /// "Unsolicited" = present in the LLM output AND in `SwissDialectForms.tokens`
    /// AND NOT present in the rules-cleaned baseline.
    ///
    /// Threshold = strict ≥1 (D-07). Aligned to the threat model: this user
    /// does not dictate Swiss German, so any unsolicited dialect form is
    /// rejected.
    ///
    /// - Parameters:
    ///   - rulesCleaned: deterministic Swift-side cleanup output (rules pass).
    ///   - llmOutput: post-stripPreamble LLM output.
    /// - Returns: `llmOutput` when delta == 0; else `rulesCleaned`.
    public static func gateLLMDialect(rulesCleaned: String,
                                      llmOutput: String) -> String {
        let baseline = Set(tokenizeForDialectGate(rulesCleaned))
        let candidate = tokenizeForDialectGate(llmOutput)
        for tok in candidate where SwissDialectForms.tokens.contains(tok) && !baseline.contains(tok) {
            #if DEBUG
            Logger(subsystem: "com.dicticus", category: "cleanup")
                .info("gateLLMDialect: demoted on token '\(tok, privacy: .public)'")
            #endif
            return rulesCleaned
        }
        return llmOutput
    }

    /// Word-level tokenization tuned for dialect-form detection.
    /// Differs from `normalizeForGate` (which collapses to a single string for
    /// distance comparison): this returns an array of lowercased word tokens
    /// suitable for set membership.
    static func tokenizeForDialectGate(_ s: String) -> [String] {
        let lowered = s.lowercased()
        // Split on Unicode whitespace + punctuation. Apostrophes within tokens
        // (e.g. "s'het") are preserved as a single token; the dialect list
        // can decide whether to include them as one entry or two.
        let separators = CharacterSet.whitespacesAndNewlines
            .union(CharacterSet(charactersIn: ".,!?;:\"„“”«»()[]{}—–-"))
        return lowered.unicodeScalars
            .split { separators.contains($0) }
            .map { String(String.UnicodeScalarView($0)) }
            .filter { !$0.isEmpty }
    }
}
```

`SwissDialectForms.tokens` should be lowercased at source (single canonical form) so the membership check is O(1) average via `Set` — recommend `Set<String>` not `[String]` for the constant. If the planner keeps `[String]` for consistency with `SwissHelvetisms.words`, wrap in a `static let tokenSet = Set(tokens.map { $0.lowercased() })` for the gate.

### Tokenization edge cases

| Edge case | Decision | Rationale |
|---|---|---|
| Case mismatch (`Siite` vs `siite`) | Lowercase both sides before compare | Dialect tokens are not proper nouns; case noise |
| Trailing punctuation (`uf,` vs `uf`) | Strip via `separators` charset | Don't make every token need 2 list entries |
| Apostrophes inside word (`s'het`, `d'Mueter`) | KEEP apostrophe in token | Swiss orthography uses apostrophe meaningfully; lets the list match exactly |
| Hyphenated words (`Velo-Pumpe`) | Split on hyphen | Existing `SwissHelvetisms.words` style is per-word |
| Numbers and currency tokens | Naturally won't match the dialect list | No special handling needed |
| Same Swiss word in raw AND llm output | Excluded from delta (set difference) | "Speaker actually said it" exception per D-07 — accept the speaker, even though we don't expect this user to produce it |
| LLM emits a dialect form that IS in `SwissHelvetisms.words` (e.g. `Velo`) | Will not trigger because `SwissDialectForms.tokens` is curated to EXCLUDE substantive helvetisms | D-06 — substantive helvetisms are an orthography concern, not a dialect concern |

## 5. 20.06 Patterns to Replicate / Pitfalls to Avoid

### Replicate: CurrencyAntiFlip architecture

Phase 20.06 Plan 02 introduced the closest architectural analog to 20.08's gate. Reusable patterns:

1. **Topic-specific token list as a separate Swift constant** — `CurrencyAntiFlip.Family` enum + word-form pattern. 20.08 follows the same shape with `SwissDialectForms.tokens` (just simpler — a flat `[String]` rather than a typed enum, because dialect forms don't have families).
2. **Graceful-degradation contract** — `CurrencyAntiFlip` documents that any unexpected input shape returns the input unchanged. 20.08's gate has a similar safety: empty input, nil tokens, or zero matches all return `llmOutput` (no demotion). Document this in the function header.
3. **Pre-LLM detection + post-LLM revert pattern** — Currency uses TWO touch points (prompt anchor + post-LLM revert). 20.08 only needs ONE (the gate); the prompt rework is the equivalent of the prompt anchor, and the gate is the equivalent of the post-LLM revert. Same defense-in-depth philosophy, simpler shape.
4. **Test-first contract lock for prompt content** — Phase 20.06-01 added `CleanupPromptTests` methods that assert literal strings in `build()` output. 20.08 should do the same for the new HELVETISMS block (e.g. `testHelvetismsBlockOutputsHighGermanDirective`, `testHelvetismsBlockNoLongerEmbedsSwissHelvetismsReferenceList` — the negative test locks the removal).

## 7. Open Questions / Risks

1. **Will the spike actually find a clear winner?** If all four variants still produce some Swiss-ification under random sampler seeds, the planner needs a fallback: ship the LEAST-leaky variant + lean harder on the gate. The gate is the safety net regardless of prompt outcome — that's why D-15 keeps Levenshtein and D-08 stacks both gates.
2. **`SwissDialectForms.tokens` initial coverage gap.** The starter list above is ~30 tokens. The 2026-04-27 failure used 8 distinct dialect forms; 6 are in the starter list, 2 (`wuer`, `de`) need adjudication (`wuer` is Swiss-specific, include it; `de` is a homograph, exclude it). Planner: review the starter list during planning and prune homographs aggressively — false positives demote LEGITIMATE LLM cleanups.
3. **iOS build status.** Phase 20.06 documented FluidAudio API drift in `IOSTranscriptionService.swift`. Status unknown as of 2026-04-27. If still broken, iOS gate-function unit tests cannot execute in this phase — would need to be deferred or fixed alongside.
4. **Spike harness lifetime.** D-Discretion lets Claude decide whether to keep behind `#if DEBUG` or remove entirely. Recommend keep — Phase 20.08 will not be the last LLM-prompt iteration, and a reusable harness pays for itself.
5. **Tokenization gap: leaked Gemma chat-template fragments.** `CleanupService.stripPreamble` (lines 448–457) has a regex that strips leaked `</start_of_turn>`/`<bos>`/`<eos>` etc. The gate operates on POST-stripPreamble output (because that's what `TextProcessingService` passes in), so this is fine — but the planner should be aware that any future change to where the gate runs in the call chain must preserve "after stripPreamble" ordering.
