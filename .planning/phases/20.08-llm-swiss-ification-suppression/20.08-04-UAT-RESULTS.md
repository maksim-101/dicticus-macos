## Findings — regressions introduced by variant (g15)

🔴 **R-G15-01 (CRITICAL, cross-platform): currency-digit truncation.**

Both macOS and iOS independently truncated `102.50 Franken` →
`12.50 Franken` (lost the leading `10` hundreds digit). Identical
mutation on both platforms confirms this is **prompt-driven**, not a
sampler fluke. The ground-truth ASR captured `102.50` correctly on
macOS plain-dictation; the LLM rewrote it under cleanup.

Hypothesis: variant (g15)'s contextual-mishear-repair duty
(VARIANT-G-RATIONALE §1 reframing — "slight speech correction with
gap-filling and contextual mishear repair") is over-firing on numbers
that *look* implausible by surrounding context (`über 99 Franken` +
colloquial `vielleicht sogar um die`) but were in fact correctly
transcribed. The 4-shot frame's third example (`1250 Franken 20` →
unchanged) trains "preserve digits in 4-digit currency", not
"preserve digits in 3-digit decimal currency".

This is the failure-mode class Phase 20.06 was created to prevent
(F-20-UAT-02 was the Franken→Euro flip; this is a sibling class —
digit mutation rather than currency-symbol flip). The Phase 20.06
STRICT speaker-explicit currency anchor was REMOVED from the German
path by the variant (g15) verbatim contract — that removal is now
implicated.

🟡 **R-G15-02 (iOS-only): language-detection bleeds into preamble.**

iOS `AI cleanup (DE)` opens with three English fragments before
falling back to German mid-sentence:
- `"And we are on the AI Cleanup Version, now auf Deutsch."`
- `"And there will be clapping."` (likely "Diese Woche" mishear)
- `"This week I have a homeoffice,"` then German continues.

variant (g15) only fires on `language == "de"`; if FluidAudio /
language-detector reports `en` for the opening clauses, the EN
`defaultInstruction` branch runs and produces these English fragments.
Probably not a g15-induced regression per se — likely an upstream ASR
language-detection issue compounded by a per-utterance language
decision rather than per-sentence — but variant (g15) gives the user
no path to recover (German cleanup never ran on the German content).

🟢 **Beneficial side-effects of variant (g15) (worth keeping):**
- macOS `Gestern muss ich` → `Gestern musste ich` (tense fix, correct).
- macOS comma added at clause boundary (`Schulferien waren, und das`).
- macOS `über das macOS App` → `über die macOS App` (gender fix).
- iOS `eine verdammt grosse Freude` preserved with `grosse` (Swiss
  orthography applied).

🟦 **Out-of-scope observation (NOT a g15 regression):**
- macOS+iOS opening: speaker said "AI cleanup", ASR captured
  "Cleanup-Version" with the "AI" prefix already missing in plain
  dictation. Word loss happened upstream of the LLM.

🟦 **Confirmed working (separate from g15):**
- The Fn hotkey combo for macOS push-to-talk is functional (carried
  over from prior phase work; user confirmed).

### What the gap-closure plan must address

R-G15-01 (currency truncation) is the hard blocker. Two candidate
fixes worth weighing in the gap-plan discussion:

1. **Restore Phase 20.06 STRICT currency anchor** to the German path
   (append after the variant g15 user turn body, before
   `<end_of_turn>`). Re-introduces explicit "preserve digits exactly"
   constraint that was lost in the verbatim contract.
2. **Add a digit-preservation example** to the variant (g15) 4-shot
   frame — e.g. a 5th `ORIGINAL/KORRIGIERT` pair with an `xxx.yy`
   decimal currency that demonstrates "don't repair plausible-looking
   numbers". Risk: bloats the prompt header (g14 was disqualified for
   header bloat — VARIANT-G-RATIONALE §10.2).

R-G15-02 (iOS language-detection bleed) is real but appears upstream
of the prompt — likely a separate phase. Worth flagging in the
gap-plan but probably not in scope.

Variant (g15)'s primary anti-Swiss-ification contract held cleanly on
both platforms — the gap-closure should additively patch the digit
preservation gap, NOT revert to variant (e) or earlier preservation-
first prompts.
