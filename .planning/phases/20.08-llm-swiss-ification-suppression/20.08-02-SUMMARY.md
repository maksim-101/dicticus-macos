## Stacking-Safety Identity Proof

When the dialect gate demotes:
1. `processedText` becomes `rulesCleanedText` (gate1 output).
2. Levenshtein gate then computes `LevenshteinDistance.normalizedDistance(rulesCleanedText, rulesCleanedText)` = `0.0`.
3. `0.0 ≤ 0.30` → gate2 returns `processedText` (i.e., `rulesCleanedText`) unchanged.
4. Final output = `rulesCleanedText`. No double-demotion artefact.

When the dialect gate passes:
1. `processedText` is the LLM output (gate1 output).
2. Levenshtein gate runs as before — its own threshold logic applies. Stacking is purely additive in terms of acceptance criteria (output is accepted only if BOTH gates accept).

Empirically validated by the R7 integration test: mock returned `"uf de andere Siite"`, dialect gate demoted (logged), Levenshtein gate ran on `(rulesCleanedText, rulesCleanedText)` → identity, final output had zero dialect tokens (verified via `tokenizeForDialectGate` word-boundary check).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] R7 substring assertion matched 'uf' inside 'auf'**
- **Found during:** Task 3 first test run.
- **Issue:** Plan-specified assertion `XCTAssertFalse(output.lowercased().contains("uf "))` fails because `"auf der..."` lowercased contains the substring `"uf "` (positions 1-3 of `"auf "`). After demotion the output equals the rules-cleaned baseline `"auf der anderen seite"`, which legitimately contains `"auf "` — but the substring check incorrectly flags it as a Swiss dialect leak.
- **Fix:** Switched the assertion to use `Set(CleanupService.tokenizeForDialectGate(output))` for word-boundary-aware token membership, which is also the same tokenizer the gate itself uses. Honours the test intent (no Swiss dialect tokens as standalone words) without false-positive substring overlap. Applied to both iOS and macOS R7 tests for parity.
- **Files modified:** iOS/DicticusTests/TextProcessingServiceTests.swift, macOS/DicticusTests/TextProcessingServiceTests.swift
