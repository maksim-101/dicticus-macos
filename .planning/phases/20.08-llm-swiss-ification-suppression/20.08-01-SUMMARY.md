# Phase 20.08 Plan 01: SwissDialectForms Data File + Cross-Platform Test Pair Summary

Curated 39-token Swiss German dialect-form list `SwissDialectForms.tokens` (distinct from `SwissHelvetisms.words`) lands in `Shared/Models/` with full XCTest data-shape coverage on macOS (6 tests passing, first such test on macOS target). iOS test file written with identical body; execution deferred behind the pre-existing FluidAudio drift documented in `deferred-items.md`.

## What Shipped

- `Shared/Models/SwissDialectForms.swift` — public enum with `tokens: [String]` (39 entries) + `licenseAttribution: String`. Mirrors `SwissHelvetisms.swift` shape; tokens lowercased at source for `Set<String>` membership lookup by Plan 02's `gateLLMDialect`.
- `iOS/DicticusTests/SwissDialectFormsTests.swift` — 6 test methods covering R4 (count-in-range, all-lowercase, unique, no-overlap-with-`SwissHelvetisms.words`, UAT-anchor canonical tokens present) + R5 (license attribution literals).
- `macOS/DicticusTests/SwissDialectFormsTests.swift` — verbatim parity twin; **first `Shared/Models/` data-shape test on the macOS target** per PATTERNS.md cross-cutting issue #1.
- `macOS/Dicticus.xcodeproj/project.pbxproj` — manually edited to wire (a) `SwissDialectForms.swift` into the Dicticus app target and (b) `SwissDialectFormsTests.swift` into the DicticusTests target, both into Models/DicticusTests groups + corresponding `PBXSourcesBuildPhase` entries.

## How It Works

Tokens are stored as a flat lowercased `[String]` (target band 30..50; final count 39). Plan 02's `gateLLMDialect` will wrap this in a `Set<String>` for O(1) membership lookup. Distinct from `SwissHelvetisms.words` (substantive helvetisms used as a prompt orthography anchor) — `SwissDialectForms.tokens` is a curated dialect-form list for *post-LLM demotion* when Gemma 4 E2B unsolicited-Swiss-ifies clean Standard High German output.

`licenseAttribution` is a runtime-assertable string (W9 lock pattern from Phase 19.5) — tests assert on the constant rather than walking `#filePath` to the source file, which keeps assertions stable across CI / shipped builds.

## Token-List Pruning Decisions

Per RESEARCH.md §3 starter list, the following candidates were excluded (not in the final 39):

| Token | Reason | Source |
|-------|--------|--------|
| `de` | Standard High German article ("der/die/das" homograph) and language code; would false-positive on every German sentence | RESEARCH.md §3 exclusion note |
| `sind` | Standard HG 3pl of "sein"; false-positive risk too high | RESEARCH.md §3 exclusion note |
| `müesli` | Loanword in Standard High German | RESEARCH.md §3 exclusion note |

Final composition (39 tokens): 9 UAT-anchor forms (uf, siite, wahrschiinli, alli, mini, wuer, het, hie, usfiltere) + 16 high-frequency Swiss particles/auxiliaries (isch, gsi, gseh, gha, nid, nüt, öppis, öpper, dini, sini, üses, iri, chönd, händ, wänd, müend) + 14 vowel-shift / consonant markers (iikaufe, usgfloge, choschtet, chauft, gässe, gschribe, gloffe, natürli, speter, chliis, chliins, beidne, biide, büechli).

Note: the plan body claimed "Final count after the list above: 38 tokens." That was an off-by-one in the plan; the skeleton block itself contains 39 entries, and 39 is fully within the validated 30..50 band.
