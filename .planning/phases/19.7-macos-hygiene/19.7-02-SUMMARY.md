# Phase 19.7 Plan 02: Input Monitoring Permission Row Summary

Add Input Monitoring as a 3rd permission dimension to PermissionManager and the menu bar dropdown — closes UAT row M3 by giving users an in-app surface for the Input Monitoring grant that ModifierHotkeyListener.flagsChanged actually requires on macOS 15+.

## What Changed

### Task 1 — PermissionManager.swift (commit `bffc315`)
- Added `import IOKit.hid` after the existing `@preconcurrency import ApplicationServices`.
- Added `@Published var inputMonitoringStatus: PermissionStatus = .pending` after `accessibilityStatus`.
- **Deleted lines 60–62** of the prior file — the misleading comment
  `/// True when both required permissions are .granted.` /
  `/// Input Monitoring is NOT needed — KeyboardShortcuts uses Carbon RegisterEventHotKey,` /
  `/// not NSEvent.addGlobalMonitorForEventsMatchingMask.` — and replaced with an accurate
  multi-line docstring explaining why Input Monitoring IS required (because
  `ModifierHotkeyListener.NSEvent.addGlobalMonitorForEvents(matching: .flagsChanged)` is
  gated by Input Monitoring on macOS 15+).
- Updated `allGranted` to require all three statuses.
- Extended `checkAll()` with the IOHIDCheckAccess switch:
  ```swift
  let hidAccess = IOHIDCheckAccess(kIOHIDRequestTypeListenEvent)
  switch hidAccess {
  case kIOHIDAccessTypeGranted: inputMonitoringStatus = .granted
  case kIOHIDAccessTypeDenied:  inputMonitoringStatus = .denied
  case kIOHIDAccessTypeUnknown: inputMonitoringStatus = .pending
  default:                      inputMonitoringStatus = .pending
  }
  ```
- Added `requestInputMonitoring()` calling `IOHIDRequestAccess(kIOHIDRequestTypeListenEvent)`.

### Task 2 — PermissionRow.swift (commit `1f0045e`)
- Replaced `Button(status == .denied ? "Open Settings" : "Grant Access")` with
  `Button(status == .denied ? "Repair in System Settings" : "Grant Access")`.
- Updated the matching docstring on line 9.
- Branch logic, controlSize, fixedSize, and the surrounding `if status != .granted` guard preserved.

### Task 3 — MenuBarView.swift (commit `b6dae87`)
- Wrapped the existing PermissionRow VStack in `if !permissionManager.allGranted { ... }`.
- Appended a third PermissionRow for Input Monitoring using `permissionManager.inputMonitoringStatus`,
  `permissionManager.requestInputMonitoring()`, and `SystemSettingsURL.inputMonitoring`.
- Moved the `Divider()` that previously followed the VStack inside the conditional block, so
  when all three permissions are granted both the rows AND the divider disappear.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `@unknown default` rejected for IOHIDAccessType**
- **Found during:** Task 1 verification build
- **Issue:** The plan instructed `@unknown default: inputMonitoringStatus = .pending` for the
  IOHIDCheckAccess switch. Swift 6 / Xcode 26 reports
  `error: switch must be exhaustive` with the note `remove '@unknown' to handle remaining values`.
  IOHIDAccessType is a non-frozen C enum — Swift requires a regular `default:` branch instead.
- **Fix:** Replaced `@unknown default:` with `default:`.
- **Files modified:** `macOS/Dicticus/Services/PermissionManager.swift`
- **Commit:** `bffc315` (folded into the same Task 1 commit)

**2. [Rule 1 — Bug] PermissionRow docstring lagged the label change**
- **Found during:** Task 2 verification (`! grep -qF '"Open Settings"'` failed)
- **Issue:** Line 9's `/// - Action button: "Grant Access" for pending, "Open Settings" for denied`
  still referenced the old label. Verification only passes if "Open Settings" appears nowhere.
- **Fix:** Updated docstring to reference "Repair in System Settings".
- **Files modified:** `macOS/Dicticus/Views/PermissionRow.swift`
- **Commit:** `1f0045e` (folded into Task 2 commit)
