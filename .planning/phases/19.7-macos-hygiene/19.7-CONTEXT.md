# Phase 19.7: macOS Hygiene - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Operational hygiene to unblock daily macOS dictation. Four UAT-routed deliverables:

- **M1** Hotkey re-authorization flow when TCC pollution / post-onboarding permission loss has broken global hotkeys
- **M2** Multi-install cleanup — dev-only install/uninstall scripts that consolidate to a canonical `/Applications/Dicticus.app`
- **M3** In-app permission status indicator covering Microphone, Accessibility, **and** Input Monitoring (the missing 3rd dimension that the existing PermissionManager comment incorrectly dismissed)
- **D1** App icon canonicalization — single 1024² source, generated macOS appiconset, same glyph/shape across iOS+macOS, fix the "no icon in Finder" regression

**Not in scope:** new features (no new hotkey types, no installer GUI, no in-app uninstall menu, no deep TCC introspection beyond `mdfind` symptom detection). iOS-side polish belongs in 19.6. CH-locale fixes belong in 19.5.

</domain>

<decisions>
## Implementation Decisions

### M2 — Multi-install Cleanup (dev-only scope)

- **D-01:** Ship two new scripts; the DMG pipeline is unchanged.
  - `scripts/install-local.sh` — for developer use during iteration:
    1. Graceful quit via `osascript -e 'tell application id "com.dicticus.macos" to quit'`, fall back to `pkill -f Dicticus.app` after 3s timeout.
    2. `mdfind "kMDItemCFBundleIdentifier == 'com.dicticus.macos'"` to enumerate every copy.
    3. Filter: exclude the canonical `/Applications/Dicticus.app` target, `.Trash`, and Time Machine snapshots.
    4. Trash the rest (use `mv` into `~/.Trash/`, do not `rm -rf` — recoverable if mistake).
    5. Copy the freshly-built `.app` to `/Applications/Dicticus.app`.
    6. Re-launch via `open -a Dicticus`.
  - `scripts/uninstall.sh` — for any user (incl. developer):
    - Removes `/Applications/Dicticus.app`, `~/Library/Preferences/com.dicticus*.plist`, LaunchAtLogin entry.
    - Prompts before deleting `~/Library/Application Support/Dicticus/` (contains the ~3.1 GB Gemma model + Parakeet ASR cache + GRDB `history.sqlite`).
    - Documents in stdout that TCC entries cannot be cleaned programmatically — points user to System Settings > Privacy.
- **D-02:** `build-dmg.sh` stays a release-only artifact. The DMG ships with the standard Applications drop-link; no first-run sweeper, no embedded uninstaller. The multi-install fix lives entirely in dev scripts because real users do not produce stale copies.
- **D-03:** README adds a "Manual uninstall" section enumerating the same paths the script touches, for users who don't want to run a script.

### M1 — Hotkey Re-Authorization Flow

- **D-04:** Detection has two layers:
  - Layer 1 — `AXIsProcessTrusted()` checked on launch and whenever the menubar dropdown opens (existing PermissionManager polling already covers the steady state).
  - Layer 2 — KeyboardShortcuts registration liveness: capture registration error from KeyboardShortcuts at startup; if AX is granted but registration returns an error (rare TCC race / hotkey conflict), surface a recoverable state, not a permission-missing state.
- **D-05:** Recovery UX — when AX is `.denied` and `hasCompletedOnboarding == true`, show a sticky red/yellow banner at the top of the menubar dropdown: **"Hotkeys disabled — Accessibility permission missing. [Repair]"**. Repair button deep-links to `x-apple.systempreferences:com.apple.preference.security?Privacy_Accessibility`, plus a one-line hint to relaunch Dicticus after granting. Banner is always visible until the broken state clears.
- **D-06:** Always-visible **"Re-register hotkeys"** button in the Hotkey settings section. Tears down + re-registers KeyboardShortcuts handlers + ModifierHotkeyListener. Cheap escape-hatch for the post-sleep / post-login race where AX is granted but hotkeys silently failed to bind.
- **D-07:** Multi-copy proactive warning — at launch, run `mdfind "kMDItemCFBundleIdentifier == 'com.dicticus.macos'"`. If results > 1, surface in menubar: **"Multiple Dicticus copies detected — may cause permission issues. [Show]"**. Click reveals the paths and points the user at `scripts/install-local.sh`.

### M3 — Permission Status Indicator

- **D-08:** Add **Input Monitoring** as a third permission row. Status check: `IOHIDCheckAccess(kIOHIDRequestTypeListenEvent)`. Required because `ModifierHotkeyListener` uses `NSEvent.addGlobalMonitorForEvents(matching: .flagsChanged)` which on macOS 15+ requires Input Monitoring. The existing `PermissionManager.allGranted` comment ("Input Monitoring is NOT needed — KeyboardShortcuts uses Carbon RegisterEventHotKey") is wrong for the modifier-listener path and **must be corrected** as part of this phase.
- **D-09:** Context-dependent action button language per row:
  - `.pending` → "Grant Access" (current behavior)
  - `.denied` → **"Repair in System Settings"**
  - `.granted` → no button
- **D-10:** Per-permission Settings deep-link URLs in `SystemSettingsURL.swift`:
  - Microphone → `x-apple.systempreferences:com.apple.preference.security?Privacy_Microphone` (already exists)
  - Accessibility → `…?Privacy_Accessibility` (already exists)
  - Input Monitoring → `…?Privacy_ListenEvent` (NEW)
- **D-11:** Hide the entire permission section when all 3 are `.granted`. Re-appears the moment any drops to non-granted. Quieter steady-state UI; loud only when action is needed.
- **D-12:** Permission rows respect the Repair banner from D-05 — they coexist; the banner is the high-level call-to-action, the rows show per-permission state.

### D1 — App Icon Canonicalization

- **D-13:** Single source of truth: **`assets/icon-master.png`** at the repo root (1024×1024 PNG). Both platforms reference it. Created in this phase; replaces the current divergent state (iOS single PNG, macOS hand-maintained 10-PNG appiconset).
- **D-14:** New script `scripts/generate-icons.sh`:
  - Reads `assets/icon-master.png`.
  - Generates the macOS appiconset (10 PNGs at 16/32/128/256/512 × 1x+2x) via `sips -z`.
  - Writes to `macOS/Dicticus/Assets.xcassets/AppIcon.appiconset/`.
  - Generated outputs are committed to git (so PR diffs catch unintended icon changes).
  - Also copies the master to `iOS/Dicticus/Assets.xcassets/AppIcon.appiconset/AppIcon.png` so iOS and macOS share the same artwork.
- **D-15:** Visual design — same glyph, same shape across iOS and macOS. Aligns with Apple's recent cross-platform icon direction (macOS 14+/iOS 18+) and Strategic Decision **S3 ("One DESIGN.md for the whole project")** from `19-UAT-FINDINGS.md`.
- **D-16:** "No icon in Finder" regression — fix-forward first:
  1. Run `scripts/generate-icons.sh` against the new master.
  2. Nuke icon caches: `killall Dock Finder; sudo mdutil -E /` and rebuild a clean DMG.
  3. If icon still missing in Finder after a fresh build, run `/gsd-debug` to isolate root cause (xcodegen miswiring, codesign strip, asset catalog corruption, LaunchServices cache staleness).
- **D-17:** README adds a one-paragraph "Updating the app icon" section pointing devs at `scripts/generate-icons.sh`.

### Sequencing Within the Phase

- **D-18:** Suggested execution order: M2 first (cuts the daily-blocker root cause), then M3 (gives users in-app diagnosis for whatever still breaks), then M1 (re-auth UX builds on the M3 row infrastructure), then D1 (cosmetic, parallel-able with any of the above). Planner is free to override if dependencies dictate.

### Claude's Discretion

- Concrete copy/wording in the Repair banner and Re-register button — keep terse, German+English (Dicticus is a German-first app).
- File-tree placement of new permission status logic (extend `PermissionManager` vs introduce a thin `MultiInstallDetector`).
- Whether to use `IOHIDCheckAccess` directly or wrap behind a small typed helper for testability.
- Trash-vs-rm semantics inside `install-local.sh` (lean toward Trash for recoverability, but if `osascript`/AppleScript-Trash is awkward, `mv ~/.Trash/` is acceptable).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- **`PermissionManager`** — Already polls every 2s, exposes `@Published` status, has `requestMicrophone()` / `requestAccessibility()`. Extend with `inputMonitoringStatus` and the corresponding request action; the polling loop already gives us live updates from System Settings changes.
- **`PermissionRow` view** — Already supports per-row label / status / grantAction / settingsURL. Adding a 3rd row is structural reuse with no new view.
- **`SystemSettingsURL`** struct — has `microphone` and `accessibility`; just adds an `inputMonitoring` constant.
- **MenuBarView `onAppear`** — already starts polling. Run the `mdfind` multi-copy check here too (D-07) — single hook, no new lifecycle.
- **`build-dmg.sh`** — codesign patterns and Developer ID flow are known-good; install-local.sh can borrow shell idioms but does NOT need to re-sign (it just moves the already-signed `.app`).

### Established Patterns

- `@MainActor`-annotated services with `@Published` state observed via `@EnvironmentObject` in views — extend, don't introduce new patterns.
- 2s polling for permission state; same cadence for liveness checks (D-04 layer 1).
- Error-logged-but-non-fatal pattern in `ModifierHotkeyListener` (the `.error("NSEvent global monitor creation FAILED")` log) — use the same shape for KeyboardShortcuts liveness errors (D-04 layer 2).
- Plain shell scripts in `scripts/` with `set -euo pipefail` and clear `=== Step N: ===` headers (cf. `build-dmg.sh`) — match the style for install/uninstall/generate-icons scripts.

### Integration Points

- **MenuBarView dropdown** — Repair banner (D-05) lives at the top, before the warmup row; multi-copy warning (D-07) lives just below the banner; existing permission rows expand to 3.
- **HotkeySettingsView** — gains a single "Re-register hotkeys" button row at the bottom of its existing layout.
- **DicticusApp `@main`** — already sets up `PermissionManager` env-object; no new wiring needed for D-08.
- **scripts/** — currently has `build-dmg.sh` and `verify-memory.sh`; new files (`install-local.sh`, `uninstall.sh`, `generate-icons.sh`) sit alongside.
- **Repo root** — new `assets/` directory introduced for `icon-master.png` (D-13).

<specifics>
## Specific Ideas

- "**Repair**" is the canonical button word for fix-broken-grant flows (vs "Grant Access" for first-time grant). Distinguishes from onboarding language and signals to the user that something *was* working before.
- The multi-copy warning is **launch-only** by design — running `mdfind` every 2s would be wasteful. Once-at-launch is enough because new copies don't appear during a single app session in the user's actual workflow.
- Trash (not `rm -rf`) for stale copies in `install-local.sh` — the dev-tooling principle is "recoverable if you misclick or if a path glob is wrong".
- Generated icon PNGs are committed (D-14) so a PR review can spot an unintended icon change visually in the diff. Auto-regen-at-build was rejected for this reason.
- Strategic Decision S3 ("one DESIGN.md") implies brand consistency; the same-icon-on-both-platforms decision (D-15) is the visible side of that.

</specifics>

<deferred>
## Deferred Ideas

- **In-app "Uninstall Dicticus…" menu entry** — considered, scoped out. Real users who want to remove the app + reclaim 3 GB of model files would benefit, but it expands the phase beyond "unblock daily dictation". Revisit when v2.2+ does a broader UX polish.
- **First-run multi-copy sweeper inside the running app** — considered, rejected. Risk of trashing legitimate side-by-side installs (beta + stable) for a problem that affects only developers. Dev-only `install-local.sh` solves the actual problem without that risk.
- **Codesign-team-ID verification before trashing stale copies** — considered, deferred. Useful belt-and-suspenders but adds shell complexity for a script only the developer runs on their own machine. Add later if mistakes happen.
- **TCC introspection** — programmatically reading TCC.db is SIP-protected and brittle across macOS versions. We rely on symptoms (AX trust check, mdfind multi-copy detection) rather than direct DB inspection.
- **Telemetry / diagnostic log capture** for failed hotkey registrations — would help debug post-ship issues, but Dicticus is fully local + no telemetry; out of scope here.
- **Auto-detect-and-recover for AX-true-but-hotkey-not-firing** — currently handled by the manual Re-register button. Could be automated via a "no events in N seconds since launch" heuristic but risks false positives (user just hasn't dictated yet).
- **macOS-HIG-distinct icon canvas** (rounded square w/ offset shadow vs iOS squircle) — considered, declined per S3. Revisit only if Apple's cross-platform design language shifts back.

</deferred>
