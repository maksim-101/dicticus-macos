# Phase 19.7: Code Review Report

**Reviewed:** 2026-04-25T07:25:24Z
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

The four sub-plans of phase 19.7 (install/uninstall scripts, Input Monitoring permission, hotkey re-auth recovery, icon pipeline) are well-structured and defensively coded. Shell scripts use `set -euo pipefail`, paths are quoted, and TCC limitations are documented honestly. Swift code respects Swift 6 concurrency (`@MainActor`, `[weak self]`, module-level cached `kAXTrustedCheckOptionPrompt`). No critical security issues were found.

The findings below are correctness / robustness concerns. The most consequential is **WR-01**: `PermissionManager.checkMultipleInstalls()` runs a synchronous `Process.waitUntilExit()` on the `@MainActor` from `MenuBarView.onAppear`, which can stall the UI if `mdfind` is slow on a freshly-booted Mac. **WR-02** flags that the repair banner copy lies when `registrationFailed` is true but Accessibility is granted. **WR-03** notes a destination-path collision in `install-local.sh` when two stale copies share a parent directory name.

A handful of minor issues (code duplication between `setup()` and `reregisterAll()`, missing `xargs`-style guard on the legacy `for f in $PREFS_GLOB` pattern, swallowed errors in `relaunchApp`) round out the list.

## Warnings

### WR-01: Synchronous `mdfind` blocks `@MainActor` at popover open

**File:** `macOS/Dicticus/Services/PermissionManager.swift:166-193`
**Issue:** `checkMultipleInstalls()` is `@MainActor`-isolated and uses `Process` with `task.waitUntilExit()` — a blocking call on the main thread. It is invoked from `MenuBarView.onAppear` (`MenuBarView.swift:210`), which runs on every popover open. `mdfind` is normally <100 ms but can take seconds on a freshly-booted Mac before Spotlight finishes indexing, or when the Spotlight daemon is busy. While it blocks, the menu bar dropdown will appear frozen.
**Fix:** Run the probe off the main actor and update `@Published` state on return. Example:

```swift
func checkMultipleInstalls() {
    Task.detached(priority: .utility) {
        let paths = Self.runMdfind()
        await MainActor.run { [weak self] in
            self?.multipleDicticusCopies = paths
        }
    }
}

private static func runMdfind() -> [URL] {
    let task = Process()
    task.executableURL = URL(fileURLWithPath: "/usr/bin/mdfind")
    task.arguments = ["kMDItemCFBundleIdentifier == 'com.dicticus.macos'"]
    let pipe = Pipe()
    task.standardOutput = pipe
    task.standardError = Pipe()
    do {
        try task.run()
        task.waitUntilExit()
        let data = pipe.fileHandleForReading.readDataToEndOfFile()
        guard let output = String(data: data, encoding: .utf8) else { return [] }
        return output
            .split(separator: "\n")
            .map(String.init)
            .filter { !$0.isEmpty && $0.hasSuffix("/Dicticus.app") }
            .map { URL(fileURLWithPath: $0) }
    } catch {
        return []
    }
}
```

### WR-02: Repair banner text is misleading when only `registrationFailed` is true

**File:** `macOS/Dicticus/Views/MenuBarView.swift:38-65`
**Issue:** The banner condition is `(accessibilityStatus == .denied && hasCompletedOnboarding) || hotkeyManager.registrationFailed`, but the body unconditionally says *"Hotkeys disabled — Accessibility permission missing."* and the action is *"Repair → open Accessibility pane."* When `registrationFailed` is true with Accessibility granted (the post-sleep / post-login race that motivated D-04 layer 2), the message points users to the wrong remedy. The actually useful action — `hotkeyManager.reregisterAll()` — is buried in `HotkeySettingsView`.
**Fix:** Branch the body on which condition triggered the banner. When `registrationFailed` is true and AX is granted, show *"Hotkeys stopped responding — try re-registering."* with a `Re-register` button calling `hotkeyManager.reregisterAll()`.

```swift
let axProblem = permissionManager.accessibilityStatus == .denied
    && permissionManager.hasCompletedOnboarding
let registrationProblem = hotkeyManager.registrationFailed && !axProblem

if axProblem || registrationProblem {
    HStack(spacing: 8) {
        Image(systemName: "exclamationmark.triangle.fill")
            .foregroundColor(.red)
        Text(axProblem
             ? "Hotkeys disabled — Accessibility permission missing."
             : "Hotkeys stopped responding.")
        Spacer()
        Button(axProblem ? "Repair" : "Re-register") {
            if axProblem {
                SystemSettingsURL.open(SystemSettingsURL.accessibility)
            } else {
                hotkeyManager.reregisterAll()
            }
        }
    }
}
```

### WR-03: `install-local.sh` trash destination can collide when stale copies share parent dir name

**File:** `scripts/install-local.sh:74-78`
**Issue:** The destination filename is built as `Dicticus-stale-$TIMESTAMP-$(basename "$(dirname "$p")").app`. If two stale copies live in directories with the same basename (e.g. `~/Downloads/Dicticus.app` and `/tmp/Downloads/Dicticus.app`, or two different Xcode `Build/Products/Release/` folders both basenamed `Release`), the second `mv` will silently move *into* the first (since the destination is a directory at that point), nesting one app inside the other. `set -e` won't catch this because `mv` returns 0.
**Fix:** Make the destination unique by appending an index, or hash the full source path:

```bash
i=0
for p in "${STALE_COPIES[@]}"; do
    i=$((i+1))
    dest="$TRASH_DIR/Dicticus-stale-$TIMESTAMP-$i-$(basename "$(dirname "$p")").app"
    [ -e "$dest" ] && dest="$TRASH_DIR/Dicticus-stale-$TIMESTAMP-$i-$RANDOM.app"
    echo "  mv $p -> $dest"
    mv "$p" "$dest"
done
```

### WR-04: `uninstall.sh` glob expansion executes if shell `nullglob` is unset

**File:** `scripts/uninstall.sh:41-45`
**Issue:** `for f in $PREFS_GLOB` relies on the glob expanding to filenames. If no `com.dicticus.*.plist` files exist, the unquoted variable expands to the literal pattern string `"$HOME/Library/Preferences/com.dicticus.*.plist"`, which fails the `[ -e "$f" ]` guard so nothing wrong happens — *today*. But this is fragile: any future maintainer who removes the `[ -e ]` guard or changes loop semantics introduces a bug. Bash's `nullglob` is also not enabled, and the glob is on a *variable* not a literal, so `set -f` interactions are subtle. Note also that `set -u` does **not** fire here because `PREFS_GLOB` is assigned, but if the variable were ever unset the script would abort at the wrong place.
**Fix:** Use `find` for unambiguous behavior, or set `shopt -s nullglob` and quote-unquote correctly:

```bash
shopt -s nullglob
prefs=("$HOME/Library/Preferences/"com.dicticus.*.plist)
shopt -u nullglob
for f in "${prefs[@]}"; do
    rm -f "$f"
    echo "  Removed $f"
done
```

### WR-05: `relaunchApp` swallows `Process.run()` errors and quits anyway

**File:** `macOS/Dicticus/Views/PermissionRow.swift:73-84`
**Issue:** `try? task.run()` discards any error from spawning `/usr/bin/open`. The 0.5 s `asyncAfter` then unconditionally calls `NSApplication.shared.terminate(nil)` regardless of whether the relaunch process actually started. If `open` fails (sandbox, missing binary, signature issue), the user is left with no running app and no diagnostic.
**Fix:** Check the result and only terminate on success:

```swift
private static func relaunchApp() {
    let url = Bundle.main.bundleURL
    let task = Process()
    task.executableURL = URL(fileURLWithPath: "/usr/bin/open")
    task.arguments = ["-n", url.path]
    do {
        try task.run()
        DispatchQueue.main.asyncAfter(deadline: .now() + 0.5) {
            NSApplication.shared.terminate(nil)
        }
    } catch {
        // Surface to the user — don't quit on failure
        NSLog("Relaunch failed: \(error)")
    }
}
```

## Info

### IN-01: `setup()` and `reregisterAll()` duplicate AsyncStream consumer setup

**File:** `macOS/Dicticus/Services/HotkeyManager.swift:101-127, 170-190`
**Issue:** The two `Task { ... for await event in KeyboardShortcuts.events(for: ...) ... }` blocks are repeated almost verbatim in `setup()` and `reregisterAll()`, with only the `hotkeyLog.info`/`hotkeyLog.error` calls dropped from the second copy. Drift between these is a future bug source (e.g., a logging change in one but not the other).
**Fix:** Extract a private helper:

```swift
private func spawnConsumers() {
    plainDictationTask?.cancel()
    plainDictationTask = Task { [weak self] in
        guard let self else { return }
        hotkeyLog.info("AsyncStream started for plainDictation")
        for await event in KeyboardShortcuts.events(for: .plainDictation) {
            switch event {
            case .keyDown: self.handleKeyDown(mode: .plain)
            case .keyUp:   self.handleKeyUp(mode: .plain)
            }
        }
        hotkeyLog.error("AsyncStream for plainDictation ENDED")
        self.registrationFailed = true
    }
    // …same for cleanupTask
}
```

Both `setup()` and `reregisterAll()` then call `spawnConsumers()`.

### IN-02: `await MainActor.run { ... }` is redundant inside a `@MainActor` class

**File:** `macOS/Dicticus/Services/HotkeyManager.swift:112, 126, 178, 189`
**Issue:** `HotkeyManager` is annotated `@MainActor` (line 15), so the `Task { [weak self] ... }` blocks inherit `@MainActor` isolation from the enclosing context. The `await MainActor.run { self.registrationFailed = true }` calls are redundant — `self.registrationFailed = true` would be valid directly. This is harmless but adds visual noise.
**Fix:** Replace `await MainActor.run { self.registrationFailed = true }` with `self.registrationFailed = true`.

### IN-03: `pkill -f "Dicticus.app"` could match unrelated processes

**File:** `scripts/install-local.sh:41`, `scripts/uninstall.sh:28`
**Issue:** `pkill -f` matches the entire command line, not the executable name. Any process whose argument list contains the substring `Dicticus.app` — including a shell session running these very scripts from a path containing that substring, or an editor open on a file under a `Dicticus.app/` directory — will be killed. The risk is small for a dev-only script but worth narrowing.
**Fix:** Prefer matching the bundle path strictly: `pkill -f "/Dicticus.app/Contents/MacOS/Dicticus"` or use `osascript` with `force quit` as a fallback.

### IN-04: `osascript` graceful-quit has no upper bound separate from the 3-second polling loop

**File:** `scripts/install-local.sh:34-43`, `scripts/uninstall.sh:22-30`
**Issue:** If the AppleScript event hangs (rare, but possible if the target app is non-responsive at the Apple Event level), `osascript` blocks the script. There is no explicit timeout on the `osascript` invocation itself — only on the *post-quit polling loop*.
**Fix:** Wrap with `timeout`-style guard, e.g. `( osascript -e ... ) & APID=$!; ( sleep 2; kill $APID 2>/dev/null ) & ; wait $APID`. Low priority — the real-world hit rate is near zero.

### IN-05: `bindState()` re-runs whenever `cleanupService` is assigned

**File:** `macOS/Dicticus/Services/HotkeyManager.swift:58-60, 193-209`
**Issue:** The `cleanupService` `didSet` calls `bindState()`, which re-creates the `Publishers.CombineLatest3` subscription. `cancellables.removeAll()` (line 194) prevents a leak, but the published `pipelineState` will briefly emit an `.idle` value even if the underlying state hasn't changed, because the new combine pipeline starts from initial values. Today this is harmless because `setup()` is the only caller and it assigns `cleanupService` once. If a future refactor swaps the cleanup service mid-flight, the UI could flicker.
**Fix:** Either (a) document that `cleanupService` is set-once after `setup()`, or (b) make the publisher path tolerant of mid-pipeline assignment (e.g., use `flatMapLatest` over a `CurrentValueSubject<CleanupService?, Never>`).

### IN-06: `generate-icons.sh` does not validate `sips` exit codes per slot

**File:** `scripts/generate-icons.sh:57-63`
**Issue:** `set -euo pipefail` is set, so a single `sips` failure aborts the script — but with `>/dev/null` redirecting stdout, the user sees no per-slot error message. A failed slot leaves the appiconset in a partially-regenerated state. Low impact since `sips` failures are rare on a working master file.
**Fix:** Capture stderr explicitly: `sips -z "$size" "$size" "$MASTER" --out "$out" >/dev/null 2>&1 || { echo "ERROR: sips failed for $filename"; exit 1; }`. Or accept the current behavior and note that Step 1's 1024x1024 verification already de-risks the most common cause.
