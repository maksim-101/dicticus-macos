## Multi-install Cleanup (M2)

### Q: How should the build/install pipeline handle the canonical app location?

| Option | Description | Selected |
|--------|-------------|----------|
| Dev script + clean DMG | `scripts/install-local.sh` for dev sweep + install; DMG ships unchanged | ✓ |
| DMG-only with first-run sweeper | Self-cleaning at first launch for every user | |
| Both | Dev script + in-app sweeper as belt-and-suspenders | |

**User's choice:** Dev script + clean DMG (Recommended)
**Notes:** User initially asked "Why do you recommend the first option here?" — response explained that the multi-install mess is a developer artifact (xcodebuild + DMG + dragged test copies + per-path TCC), not a user-time one. Putting a sweeper inside the running app risks trashing legitimate side-by-side installs and adds startup cost for the 99% who don't need it. User accepted the recommendation after explanation.

### Q: How wide should the stale-copy scan reach?

| Option | Description | Selected |
|--------|-------------|----------|
| Spotlight mdfind by bundle ID | `mdfind "kMDItemCFBundleIdentifier == 'com.dicticus.macos'"` system-wide, filter canonical/Trash/TimeMachine | ✓ |
| Hardcoded path list | ~/Downloads, ~/Desktop, ~/Applications, macOS/build/ | |
| Spotlight + verify bundle identity | mdfind + per-hit codesign team-ID check | |

**User's choice:** Spotlight mdfind by bundle ID (Recommended)
**Notes:** Codesign verification deferred — see CONTEXT.md deferred section.

### Q: Where should the uninstaller live and what should it remove?

| Option | Description | Selected |
|--------|-------------|----------|
| Script + in-app menu entry | scripts/uninstall.sh + menubar "Uninstall Dicticus…" | |
| Script only, no in-app menu | scripts/uninstall.sh; documented in README | ✓ |
| Script-only, conservative removal | .app + plist only; leave models/history alone | |

**User's choice:** Dev script only — document user steps in README (Recommended after follow-up)
**Notes:** User initially asked "actually this issue is only a concern for myself as the developer right? If anybody else were to download and install the app, they wouldn't have the same problem, or would they?" — response distinguished multi-install (dev-only) from uninstall (dual-use because trashing the .app leaves ~3.1 GB of Gemma + ASR cache + history.sqlite orphaned). Phase 19.7 is scoped to "unblock daily macOS dictation" so we tightened to dev-script-only with README docs for users. Broader in-app uninstall menu deferred.

### Q: How should running stale instances be handled?

| Option | Description | Selected |

| Refuse to install if running | Abort with clear message; user quits manually first | |
| Skip running instance, sweep only stopped copies | Leave running alone; user reruns after quitting | |

**User's choice:** Force-quit by bundle ID, then sweep (Recommended)

## Permission Status Indicator (M3)

### Q: Which permission rows should the menubar surface?

| Option | Description | Selected |
|--------|-------------|----------|
| Mic + Accessibility + Input Monitoring | 3rd row via IOHIDCheckAccess; correct the wrong PermissionManager comment | ✓ |
| Mic + Accessibility (status quo) | 2 rows; M1 daily-blocker root cause partially survives | |
| All 3, but only show non-granted rows | Collapses when steady; harder to diagnose | |

**User's choice:** Mic + Accessibility + Input Monitoring (Recommended)
**Notes:** Pre-question investigation: ModifierHotkeyListener uses `NSEvent.addGlobalMonitorForEvents(matching: .flagsChanged)` which on macOS 15+ requires Input Monitoring. The existing comment in PermissionManager ("Input Monitoring is NOT needed — KeyboardShortcuts uses Carbon RegisterEventHotKey") is wrong for the modifier-listener path and **must be corrected** as part of this phase.
