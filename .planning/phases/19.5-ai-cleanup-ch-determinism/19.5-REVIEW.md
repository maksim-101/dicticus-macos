# Phase 19.5: Code Review Report

**Reviewed:** 2026-04-26T04:55:52Z
**Depth:** standard
**Files Reviewed:** 15

## Summary

Phase 19.5 lands the CH-determinism polish set: Swiss-default migration, Helvetism prompt block, currency anti-flip, and Swiss number formatter — plus their integration into `CleanupPrompt`/`CleanupService` and the macOS/iOS UI surfaces.

Overall the phase is in good shape:

- The migration's idempotency contract (`swissDefaultMigratedV2_1`) and respect-explicit-OFF semantics are correct and well-tested in `SettingsToggleTests`.
- The B3 parser-ordering fix (pre-classify by punctuation, never speculatively `parseSwiss` first) is implemented as documented and locked by `testGermanThousandsToSwissApostrophe`.
- The W7 currency-glyph strip-and-reattach in `SwissNumberFormatter.reformatToken` is correctly anchored on `Set<Character>` and only fires when the glyph is leading and not the only character.
- `CurrencyAntiFlip.revertCurrencyFlip` reverse-walks output tokens and uses `rangeOfMatch(... occurrence: index)` against the mutated `result` — this is range-stable because later positions are mutated first, leaving earlier `occurrence: index` lookups valid.
- AppGroup suite (`group.com.dicticus`) is consistently used in all read/write call sites for `useSwissGerman`.
- `CleanupService.cleanup` preserves the non-throwing/graceful-degradation contract throughout the new revert + format passes.
- The B5 `Family.allCases.count == 4` and W9 `licenseAttribution` runtime constants are exposed publicly and tested — they will fail loudly on the regressions they claim to lock.

The findings below are concentrated in two areas: (1) the `\b...\b` regex in `CurrencyAntiFlip.pattern` does not correctly bound non-word currency tokens (`€`, `$`, `£`, `Fr.`), which is a real functional gap on glyph- and abbreviation-form input; and (2) several minor robustness polish items on the formatter's tokenization and post-LLM toggle re-read.

## Warnings

### WR-01: `\b...\b` regex fails to match non-word currency tokens (`€`, `$`, `£`, `Fr.`)

**File:** `Shared/Utilities/CurrencyAntiFlip.swift:96`

**Issue:** The pattern `\b(CHF|Franken|Fr\.|Rappen|EUR|Euro|€|USD|Dollar|\$|GBP|Pfund|£)\b` uses ASCII word boundaries on both sides, but `€`, `$`, `£`, and the trailing `.` of `Fr.` are non-word characters. `\b` requires a `\w`↔`\W` transition, so for any of these alternatives the trailing `\b` cannot match unless the character immediately following is a word character. In practice this means:

- `"Ich habe 5€"` — match fails (`\b` after `€` requires next char to be `\w`; end-of-string is not `\w`).
- `"5 € heute"` — match fails (`\W\W` adjacency around `€`).
- `"Das kostet 5 Fr."` — match fails (the `.` is `\W`, end-of-string after it is also `\W`).
- `"$1250"` — leading `\b` requires preceding `\w`, fails at start-of-string.

Glyph-form and `Fr.` utterances therefore never enter `detectCurrencies`, which means: (a) the D-B1b STRICT prompt anchor in `CleanupPrompt.build` won't list them, and (b) `revertCurrencyFlip` cannot revert a flipped glyph. The functional fallout is bounded — Gemma typically rewrites glyphs as words, and dictation usually produces "Euro"/"Dollar"/"Franken" rather than glyphs — but it's a real gap and silently inconsistent with the prompt's "exactly as written" promise when the user dictates a glyph or `Fr.`.

**Fix:** Replace `\b...\b` with explicit non-word-or-anchor boundaries. One concise option:

```swift
private static let pattern =
    #"(?<![\p{L}\p{N}])(CHF|Franken|Fr\.|Rappen|EUR|Euro|€|USD|Dollar|\$|GBP|Pfund|£)(?![\p{L}\p{N}])"#
```

The lookbehind/lookahead approach treats "boundary" as "not adjacent to a letter or digit" rather than the `\w`↔`\W` flip, which is what we actually want for mixed word/glyph alternatives. Add fixture cases for `"5€"`, `"5 € heute"`, `"$1250"`, and `"Das kostet 5 Fr."` to lock the new behavior.

### WR-02: `Decimal(string:locale:)` parsing of separator-free decimals under `de_DE` is loosely defined

**File:** `Shared/Utilities/SwissNumberFormatter.swift:119-123`

**Issue:** `parseGerman("5.70")` calls `Decimal(string: "5.70", locale: Locale(identifier: "de_DE"))`. Under `de_DE`, `.` is the grouping separator and `,` is the decimal separator. Foundation's behavior for grouping-separator-positioned input that does not satisfy German thousands rules (3 digits between separators) is undocumented and has changed across iOS/macOS versions:

- On some versions it returns `nil` (input invalid → falls through to `parseSwiss`, which is the desired path).
- On other versions it accepts the input loosely and parses `5.70` as `570` (period read as thousands separator without enforcement).

The code's stated intent (line 92) is "fall back to parseSwiss" for inputs like `"5.70"`, but if `parseGerman` returns `570` instead of `nil`, the formatter emits `"570"` instead of `"5.70"` and `fractionDigitCount` returns 0 — silently corrupting valid Swiss decimal input.

`testSwissPeriodDecimalRoundtrip` covers this case and would fail loudly on a regression, but only on the test runner's current Foundation version. CI on a different OS could pass while production silently corrupts.

**Fix:** Make `parseGerman` strict: require a comma if a decimal is present, and reject inputs whose grouping separator pattern is not strict 3-digit. A pragmatic implementation:

```swift
private static func parseGerman(_ s: String) -> Decimal? {
    // German requires comma for decimal. If there's no comma, the input is
    // either a pure integer or a thousands-grouped integer — verify the
    // grouping-separator positions are strictly 3-digit before accepting.
    if s.contains(",") {
        return Decimal(string: s, locale: Locale(identifier: "de_DE"))
    }
    // No comma: only accept if `.` separators (if any) sit at strict 3-digit
    // positions counting from the right.
    guard isStrictGermanThousands(s) else { return nil }
    return Decimal(string: s, locale: Locale(identifier: "de_DE"))
}
```

Add a fixture case for `"5.70"` as a standalone token (already present) plus an explicit assertion that `parseGerman("5.70")` returns nil.

### WR-03: Toggle re-read in `cleanup()` can disagree with the prompt's gating

**File:** `Shared/Services/CleanupService.swift:174, 232-240`

**Issue:** `CleanupPrompt.build(...)` (line 174) reads `useSwissGerman` once to decide whether to include the Swiss STYLE/HELVETISMS blocks. After inference completes (line 232), the AppGroup suite is read again to decide whether to apply `applySwissITN` and `SwissNumberFormatter.format`. If the user toggles the value during the 0.5–8 s inference window:

- ON → OFF mid-inference: prompt instructed Swiss output, post-pass does NOT apply formatter → output has Swiss-styled LLM rewriting but German thousands/decimal punctuation if the LLM emitted them, and no apostrophe normalization.
- OFF → ON mid-inference: prompt did not instruct Swiss output, post-pass applies the apostrophe + period formatter → mixed-style output.

The window is short and the consequence is cosmetic (one botched dictation), but the asymmetry violates the principle of "one consistent gating decision per cleanup call".

**Fix:** Snapshot the toggle once at the top of `cleanup()` and pass it into both the prompt builder (as an explicit parameter) and the post-pass:

```swift
let swissDefaults = UserDefaults(suiteName: "group.com.dicticus") ?? .standard
let useSwissGerman = swissDefaults.bool(forKey: "useSwissGerman")
let prompt = CleanupPrompt.build(
    text: text,
    language: language,
    dictionaryContext: dictionaryContext,
    useSwissGerman: useSwissGerman
)
// ... inference ...
if useSwissGerman {
    cleaned = ITNUtility.applySwissITN(to: cleaned)
    cleaned = SwissNumberFormatter.format(cleaned)
}
```

`CleanupPrompt.build` then takes the boolean explicitly instead of reading defaults itself, which also makes the prompt builder pure and unit-testable without UserDefaults setup.

### IN-01: `format()` only splits on ASCII space — newline/tab tokens flow through unchanged

**File:** `Shared/Utilities/SwissNumberFormatter.swift:42`

**Issue:** `text.split(separator: " ", ...)` treats `\n`, `\t`, `\u{00A0}` (non-breaking space), and other whitespace as part of the token, not as separators. A two-line Gemma output like `"Das kostet\n1250 Franken"` would have `"kostet\n1250"` as a single token; `parseGerman` and `parseSwiss` would both fail and the token emits verbatim, missing the apostrophe normalization. Practical risk is low (Gemma's dictation cleanup output is typically single-line), but it's a silent gap.

**Fix:** Tokenize on `CharacterSet.whitespaces` instead, preserving the whitespace verbatim by using a regex split that captures the separators, or normalize whitespace before formatting. Cheapest patch: pre-replace `\n`/`\t` with a sentinel, format, then restore.

### IN-02: Tail-stripping does not handle wrapping punctuation pairs

**File:** `Shared/Utilities/SwissNumberFormatter.swift:75-82`

**Issue:** `reformatToken` strips a single trailing `.,;:?!` but does not strip leading `(`/`"`/`«`/`„` or trailing `)`/`"`/`»`/`"`. Tokens like `"(1250)"` or `"„1250"` are emitted verbatim because the parser fails on the surrounding glyph.

**Fix:** Extend tail/head detachment to a small set of paired punctuation characters, mirroring the `leadingGlyphs` pattern used for currency. Or document that wrapped tokens are intentionally out of scope.

### IN-03: `fractionDigitCount` heuristic of "3 digits after period = thousands" is correct but undocumented for ambiguous inputs

**File:** `Shared/Utilities/SwissNumberFormatter.swift:149-172`

**Issue:** Input `"1.234"` always returns 0 fraction digits (treated as German thousands → emits `"1'234"`). This is the documented Swiss preference but means a user dictating an actual decimal `1.234` (e.g., a measurement) will see it reformatted as `1'234`. The behavior is correct per D-C2's "Swiss prefers apostrophe" stance, but worth a code comment for the next reader who sees `1.234 → 1'234` in the wild and assumes a bug.

**Fix:** Add a one-line comment in `fractionDigitCount`:

```swift
// NOTE: 3-digits-after-period is treated as thousands per D-C2 — this
// intentionally favors the Swiss thousands interpretation over a rare
// decimal reading. A user who dictates "ein punkt zwei drei vier" will
// see "1'234"; this is the phase-19.5 product decision.
```

### IN-04: `SwissDefaultMigration` is called from three sites without explicit ordering guarantee

**File:** `Shared/Utilities/SwissDefaultMigration.swift:24-37`, call sites at `macOS/Dicticus/DicticusApp.swift:49`, `iOS/Dicticus/DicticusApp.swift:39, 55`, `macOS/Dicticus/Views/SwissGermanToggleRow.swift:22`

**Issue:** The migration is intentionally called from `MenuBarExtra.task`, `ContentView.onAppear`, `scenePhase == .active`, and `SwissGermanToggleRow.currentValue()` to defend against race conditions on first launch. The function itself is idempotent (gated on `swissDefaultMigratedV2_1`), so duplicate calls are safe — but the check-then-set is not atomic across threads. If two SwiftUI invocations on the same first launch happen to interleave between the `guard !suite.bool(forKey: migratedKey)` and the two `set` calls, both can write `true` to both keys. The writes are idempotent (same value), so the only observable effect is two writes instead of one — benign. Still, document this explicitly so a future reader doesn't try to "fix" the perceived race.

**Fix:** Add a short comment to `runIfNeeded()`:

```swift
// Note: The check-then-set is not atomic, but both writes are idempotent
// (always `true`), so a duplicate invocation from a parallel call site
// produces the same end state. No mutex needed.
```
