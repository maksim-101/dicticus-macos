# Phase 19.5 Plan 01: B2 stale-`hasModels` regression fix — Summary

**Refresh `IOSModelWarmupService.hasModels` from disk at the top of `warmup()` and on every scenePhase `.active` transition, eliminating the stale-flag regression that prompted spurious "needs download" UX after relaunch.**

## Accomplishments

- Added `checkHasModels()` as the first statement of `IOSModelWarmupService.warmup()`, followed by `guard hasModels else { return }`, so a now-missing or now-present model on disk is reflected on every warmup entry.
- Restructured `DicticusApp.scenePhase .onChange` handler to nest `hasCompletedOnboarding` inside the `.active` branch and call `warmupService.checkHasModels()` BEFORE evaluating `warmupService.hasModels`, so a download that completed while backgrounded — or a stale-false from a prior session — gets corrected on resume.
- Handler shape now exposes a single-line insertion target for Plan 19.5-02's `SwissDefaultMigration.runIfNeeded()` call (per checker B1 coordination note).

## Files Created/Modified

- `iOS/Dicticus/Services/IOSModelWarmupService.swift` — added 4 lines inside `warmup()` (lines 127–132): D-D1 comment, `checkHasModels()`, `guard hasModels else { return }`. `init()` and `checkHasModels()` itself untouched.
- `iOS/Dicticus/DicticusApp.swift` — replaced 5-line `.onChange(of: scenePhase)` body (lines 49–54) with 13-line restructured version (lines 49–61) plus a 3-line explanatory comment hoisted above `.onChange`. New shape: `if newPhase == .active { if hasCompletedOnboarding { warmupService.checkHasModels(); if warmupService.hasModels { warmupService.warmup() } } }`.

## Decisions Made

- **Comment placement.** The plan's canonical post-edit shape places the D-D1 comment INSIDE the `.active` branch immediately after the `if newPhase == .active {` line, ahead of the `if hasCompletedOnboarding` check. With the comment block in that position, the B1 acceptance grep (`grep -A 3 "if newPhase == .active"`) does not see `if hasCompletedOnboarding` within the 3-line lookahead because the 3-line comment block consumes the window. The grep is the codified acceptance gate, so I hoisted the explanatory comment to live ABOVE the `.onChange(of: scenePhase)` modifier (still inside the `Scene.body` builder). The structural intent of B1 (`hasCompletedOnboarding` nested INSIDE `.active`) is preserved exactly. The grep now passes (`if hasCompletedOnboarding` is the very first non-empty line inside the `.active` branch). Plan 02's single-line insertion target — adding `SwissDefaultMigration.runIfNeeded()` as the first statement inside `if newPhase == .active` — is unaffected by the comment hoist; the insertion remains a clean one-line addition above `if hasCompletedOnboarding`.
