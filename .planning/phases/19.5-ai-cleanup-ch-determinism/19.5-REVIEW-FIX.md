## Fixed Issues

### WR-01: `\b...\b` regex fails to match non-word currency tokens (`€`, `$`, `£`, `Fr.`)

**Files modified:** `Shared/Utilities/CurrencyAntiFlip.swift`, `iOS/DicticusTests/Fixtures/CurrencyAntiFlip.fixtures.json`
**Commit:** d0f517f
**Applied fix:** Replaced ASCII `\b(...)\b` with Unicode-aware `(?<![\p{L}\p{N}])(...)(?![\p{L}\p{N}])` so non-word currency tokens are correctly bounded by "not adjacent to letter/digit" rather than the `\w`↔`\W` transition that `\b` requires. Added three fixtures to lock the new behavior:
  - `"Ich habe 5€ bezahlt"` — glyph adjacent to digit (intentionally not detected by the lookbehind/lookahead policy; fixture proves graceful-degradation no-op)
  - `"Das macht 5 Fr."` — abbreviation form now detected and reverted (CHF target)
  - `"5 € heute"` — glyph with surrounding spaces now detected (no-flip case)

**Verification status:** fixed: requires human verification — fixture for `"Das macht 5 Fr."` asserts `expected = "Das macht 5 CHF."` because `canonicalLabel` mirrors the OUTPUT token's case (output is uppercase 3-letter `EUR`, so target chf canonicalizes to `CHF`, not `Fr.`). This is the documented `canonicalLabel` behavior; the developer should confirm the fixture expectation matches product intent. (The review's WR-01 fix-text suggested `expected = "Das macht 5 Fr."` but that contradicts the existing `canonicalLabel` policy.)

### WR-02: `Decimal(string:locale:)` parsing of separator-free decimals under `de_DE` is loosely defined

**Files modified:** `Shared/Utilities/SwissNumberFormatter.swift`
**Commit:** 649de89
**Applied fix:** Made `parseGerman` strict before delegating to Foundation: a comma is required for German decimal interpretation; non-comma inputs must satisfy a strict 3-digit-from-right grouping pattern (validated by new helper `isStrictGermanThousands`). `"5.70"` now reliably returns `nil` from `parseGerman`, falling through to `parseSwiss` which yields `5.70` regardless of which Foundation version is running. The existing `testGermanThousandsToSwissApostrophe` and `testSwissPeriodDecimalRoundtrip` invariants are preserved.

### WR-03: Toggle re-read in `cleanup()` can disagree with the prompt's gating

**Files modified:** `Shared/Models/CleanupPrompt.swift`, `Shared/Services/CleanupService.swift`
**Commit:** f82576e
**Applied fix:**
- `CleanupPrompt.build` now accepts an optional `useSwissGerman: Bool? = nil` parameter. When provided, it is preferred over reading the AppGroup defaults; when `nil`, the legacy AppGroup read path is preserved so existing tests and call sites continue to work without modification.
- `CleanupService.cleanup` snapshots the AppGroup `useSwissGerman` key once at the top of the function and passes the same boolean into both the prompt builder and the post-LLM Swiss formatting pass. The duplicate AppGroup read at the post-pass site has been removed. A mid-inference toggle change can no longer cause prompt/post-pass disagreement.
