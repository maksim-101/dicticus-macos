# Phase 17.5: Competitor Research -- iOS Dictation App "Return to Previous App" Patterns

**Researched:** 2026-04-23
**Domain:** iOS dictation UX, keyboard extensions, Shortcuts integration, app-switching patterns
**Purpose:** Understand how competing iOS dictation apps handle the dictation-to-clipboard-to-return flow

---

## Executive Summary

The "return to previous app" problem is **unsolved at the platform level** on iOS. No app has a clean, universal solution. The industry has converged on three main architectural approaches, each with distinct tradeoffs:

1. **Keyboard Extension (direct insert):** Text goes directly into the cursor position via `textDocumentProxy`. No app-switching needed. But microphone access requires either opening the main app briefly or maintaining a "warm" background session. Best UX when it works.

2. **Standalone App + Clipboard:** User dictates in the app, text copies to clipboard, user manually returns (swipe/home) and pastes. Simplest to build, worst UX friction. This is what Google Eloquent and most note-taking dictation apps do.

3. **Shortcuts Integration + Action Button:** Shortcut triggers dictation, copies to clipboard, but *cannot programmatically return to the previous app*. The user must swipe back or tap back. Some apps (Wispr Flow) guide the user with visual cues.

**The keyboard extension approach is the only one that avoids the return-to-app problem entirely**, because text is inserted in-place. The tradeoff is the complexity of handling microphone access from an extension.

## Competitor Analysis

### 1. Google Eloquent (Google AI Edge)

**Source:** [App Store listing](https://apps.apple.com/us/app/google-ai-edge-eloquent/id6756505519), [TechCrunch](https://techcrunch.com/2026/04/07/google-quietly-releases-an-offline-first-ai-dictation-app-on-ios/), [gHacks](https://www.ghacks.net/2026/04/08/google-launches-ai-edge-eloquent-dictation-app-on-ios-with-offline-transcription-and-filler-word-removal/)

**Architecture:** Standalone app only (no keyboard extension on iOS yet)

**Workflow:**
1. Open the Eloquent app
2. Tap record, speak
3. App transcribes locally using Gemma-based ASR models
4. Automatically removes filler words ("um", "uh") and polishes text
5. Offers reformatting options: "Key points", "Formal", "Short", "Long"
6. User taps copy -- text goes to clipboard
7. User manually switches back to their previous app and pastes

**Return-to-App Approach:** None. Pure manual clipboard-and-switch. The user must leave Eloquent manually.

**Key Innovation:** 
- Automatic filler word removal and text polishing as a post-processing step
- Custom vocabulary via Gmail sync or manual entry
- Completely free, unlimited usage, fully offline after model download
- Also supports uploading pre-recorded audio/video files from Photos or Files

**iOS Keyboard Extension Status:** Not available on iOS. The Android version will support system-wide keyboard mode, but this is explicitly Android-only for now. [VERIFIED: App Store listing and gHacks article]

**Relevance to Dicticus:** Eloquent validates the standalone-app-plus-clipboard model as acceptable for a v1. Their post-processing (filler removal, reformatting) is a feature Dicticus already has via LLM cleanup. Eloquent's lack of a keyboard extension on iOS confirms this is genuinely hard -- even Google launched without one.

### 2. KeyVox (macmixing)

**Source:** [GitHub repo](https://github.com/macmixing/keyvox/), [App Store listing](https://apps.apple.com/us/app/keyvox-ai-voice-keyboard/id6760396964)

**Architecture:** Custom keyboard extension + host app (open source, though iOS source is in the repo not fully exposed)

**Workflow:**
1. User is in any app with a text field
2. Switches to KeyVox keyboard via globe icon
3. Taps mic button on the keyboard
4. Keyboard extension checks if IPC session is "warm" (main app running in background)
5. If warm: sends start command via Darwin notifications (CFNotificationCenter)
6. If cold: opens the main KeyVox app to activate microphone
7. Main app records audio, runs ASR (Whisper or Parakeet), sends transcription back
8. Keyboard extension receives transcription via Darwin notification + shared storage
9. Text is inserted directly at cursor via `textDocumentProxy`

**Return-to-App Approach:** The keyboard extension approach **avoids the problem entirely** -- the user never leaves their current app (when the session is warm). When cold, the app briefly opens then the user returns to continue.

**Key Technical Details (from source code):** [VERIFIED: GitHub source code]
- **IPC Mechanism:** Darwin notifications via `CFNotificationCenterGetDarwinNotifyCenter()` for signaling between keyboard extension and main app
- **Shared Data:** `KeyVoxIPCBridge` manages shared state (likely via App Groups shared container)
- **State Machine:** Idle -> WaitingForApp (with timeout) -> Recording -> Transcribing -> Idle with text inserted
- **Notification Types:** `recordingStarted`, `transcribingStarted`, `transcriptionReady`, `noSpeech`
- **Cold Start Handling:** `openContainingApp(startRecordingURL)` when session is not warm
- **Models:** Two built-in engines -- Whisper and Parakeet (user-selectable)
- **Requirement:** iOS 18.6+

**Source Code Structure:**
```
iOS/KeyVox Keyboard/
  Core/
    Dictation/
      AudioIndicatorDriver.swift      -- audio feedback/visual indicators
      KeyboardCallObserver.swift      -- monitors call state
      KeyboardDictationController.swift -- main dictation orchestrator
    Transport/
      KeyboardIPCManager.swift        -- Darwin notification IPC
      KeyboardTTSController.swift     -- text-to-speech
      KeyboardTransportDisplayState.swift
    Input/   -- keyboard input handling
    Text/    -- text insertion/manipulation
    Feedback/ -- user feedback
  Views/
  App/
```

**Relevance to Dicticus:** KeyVox is the closest architectural match to what Dicticus needs. Their Darwin notification IPC pattern between keyboard extension and main app is the proven approach for handling microphone access. The "warm session" concept is key -- if the main app is running in the background, dictation feels seamless. The cold-start path (briefly opening the main app) is the fallback.

### 3. Wispr Flow

**Source:** [Setup docs](https://docs.wisprflow.ai/articles/7453988911-set-up-the-flow-keyboard-on-iphone), [Shortcuts docs](https://docs.wisprflow.ai/articles/1986921789-how-to-set-up-flow-shortcuts-for-iphone), [Paste troubleshooting](https://docs.wisprflow.ai/articles/7971211038-fix-text-not-pasting-after-dictation)

**Architecture:** Dual-mode -- keyboard extension AND Shortcuts integration

**Keyboard Extension Workflow:**
1. User switches to Flow keyboard via globe icon
2. Taps mic button
3. iOS shows "Apple now requires this to activate the microphone" screen
4. User swipes right (or waits for animation) to grant mic access
5. Returns to their app -- dictation begins
6. Transcribed text auto-inserts directly into text field via keyboard extension
7. Audio is sent to Wispr servers for transcription (NOT on-device)

**Shortcuts/Action Button Workflow:**
1. Action Button press triggers "Quick Dictation to Clipboard" shortcut
2. Wispr Flow app opens, records
3. Transcription completes, text copied to clipboard
4. User must swipe back to previous app and paste
5. On iOS 26.4+, a "swipe-back requirement" guidance screen appears

**Return-to-App Approach:**
- **Keyboard mode:** No return needed -- text inserts in-place. But the mic activation step briefly shows a system screen, which Flow redesigned with "clearer instructions" and a "purple indicator" guiding the swipe-back gesture.
- **Action Button mode:** No auto-return. Text goes to clipboard. User must manually return. Flow also saves to "Flow Notes" as a safety net.

**Key Technical Details:** [VERIFIED: Wispr Flow official docs]
- **Cloud-dependent:** Audio goes to servers (not on-device). This is a fundamental difference from Dicticus.
- **Full Access required:** "Without Full Access, Flow cannot transcribe your speech"
- **Background session:** "iOS requires Flow to have an active session to access the microphone. Open the Flow app once after restarting your phone, and Flow stays active in the background."
- **Direct text insertion (keyboard mode):** "delivers transcribed text directly in the text field without using the clipboard"
- **Failure mode:** "If the keyboard is dismissed while Flow is still processing your speech, the transcript may be lost"
- **Restricted apps:** Banking, corporate, password manager apps block third-party keyboards entirely
- **Auto-deactivation:** Flow automatically deactivates in numeric fields, phone pad fields, email fields

**Workarounds for Restricted Apps:**
- **Action Button:** "Press and hold the Action Button, then release to start dictation... text is saved to your clipboard so you can paste it anywhere"
- **Back Tap** (older models): "Double-tap or triple-tap the back of your phone, dictate, then paste into a text field"

**Relevance to Dicticus:** Wispr Flow's dual-mode approach (keyboard + Shortcuts) is the most comprehensive. The keyboard extension handles the common case (text at cursor), while the Action Button shortcut handles apps that block custom keyboards. The "warm background session" requirement matches KeyVox's pattern. The key difference is Flow is cloud-based; Dicticus would need the same UX with on-device processing.

## Pattern Analysis: How Apps Handle "Return to Previous App"

### Pattern A: Keyboard Extension (Avoids the Problem)

**Used by:** KeyVox, Wispr Flow, DictaFlow, Diction, Speechkeys, Willow, Typeless

**How it works:** Text is inserted directly at the cursor position in the host app via `textDocumentProxy.insertText()`. The user never leaves their current app.

**The Microphone Problem:** iOS keyboard extensions cannot access the microphone directly. Three sub-patterns exist:

| Sub-Pattern | How Mic Access Works | Apps Using It | UX Impact |
|-------------|---------------------|---------------|-----------|
| **Warm Background Session** | Main app runs in background, handles audio via IPC. Extension sends start/stop commands via Darwin notifications. | KeyVox | Seamless when warm. Brief app open on cold start. |
| **System Mic Activation Screen** | iOS shows a system-level "activate microphone" prompt. User swipes to confirm, returns to app. | Wispr Flow | 1-2 second interruption on first use per session. |
| **Network-Based ASR** | Audio captured somehow and sent to server; or extension triggers main app which handles audio. | DictaFlow, Diction (cloud mode) | Depends on implementation. |

**The Warm Session Pattern (KeyVox)** is the most relevant to Dicticus because it enables fully on-device processing without network dependency.

### Pattern B: Standalone App + Clipboard (Manual Return)

**Used by:** Google Eloquent, VoiceNotes, Dictly

**How it works:** 
1. User opens dictation app
2. Dictates
3. Text copied to clipboard (sometimes automatically)
4. User manually returns to previous app (swipe up, swipe back, or tap)
5. User pastes

**Return mechanism:** None. Fully manual. User friction is 2-3 gestures (home/swipe + navigate + paste).

### Pattern C: Shortcuts + Action Button (Semi-Automated)

**Used by:** Wispr Flow (secondary mode), community-built shortcuts

**How it works:**
1. Action Button triggers a Shortcut
2. Shortcut activates dictation (either native "Dictate Text" action or opens an app)
3. Transcription completes, text copied to clipboard
4. User must manually return to previous app

**Critical limitation (confirmed by multiple sources):**
> "If running a shortcut in the background, there is unfortunately no way to go back to the previous app you were in, other than opening the app again from the shortcut."
[VERIFIED: Sindresorhus Actions app documentation]

**x-callback-url partial workaround:** If the *calling app* initiates the shortcut via x-callback-url with an `x-success` parameter, iOS can return to that app after the shortcut completes. But this requires the calling app to support x-callback-url, which arbitrary apps do not.

**Community patterns:**
- "Dictate a Note" shortcut: Dictate -> save to app (Bear/Drafts) -> copy to clipboard as safety net. No auto-return. [VERIFIED: heydingus.net blog]
- "Quick Dictate" (Matthew Cassinelli): Dictate -> copy to clipboard -> user pastes manually. [VERIFIED: matthewcassinelli.com]
- Wispr Flow + Drafts: Action Button -> dictate -> text saved to Drafts AND copied to clipboard. No auto-return. [VERIFIED: Drafts community forum]

## The "Text at Cursor" Problem on iOS

### Why It's Hard

iOS does not provide a public API for apps to:
1. Detect which app/text field is currently focused
2. Insert text into another app's text field
3. Programmatically return to the previously-active app
4. Determine the frontmost app from a Shortcut or extension

### Available Mechanisms

| Mechanism | How | Limitation |
|-----------|-----|------------|
| **Keyboard extension `textDocumentProxy`** | `insertText(_:)` directly into focused field | Only works from a keyboard extension; requires keyboard to be active; no mic access |
| **Clipboard + paste gesture** | Copy to clipboard, user manually pastes | Requires user action; 2-3 gesture overhead |
| **Shortcuts "Set Clipboard" + "Open App"** | Shortcut copies text and opens a specific app | Must know which app to open; can't detect previous app |
| **x-callback-url** | App A calls shortcut with x-success URL back to App A | Only works if App A initiates the flow; not useful for Action Button triggers |
| **Share Sheet** | Run shortcut from share sheet context | Only works with selected text; can't insert text back |

### The Only Real Solution: Keyboard Extension

The keyboard extension is the **only** mechanism on iOS that can insert text at the cursor position in any app without the user leaving that app. Every serious dictation app that claims "works in every app" uses a keyboard extension.

## Recommended Architecture for Dicticus

Based on competitor analysis, the optimal architecture is a **dual-mode approach** (similar to Wispr Flow, but fully on-device):

### Primary: Keyboard Extension with IPC (KeyVox Pattern)

```
User in any app -> switches to Dicticus keyboard -> taps mic
  |
  v
Keyboard extension checks if main app session is warm
  |
  +--> Warm: Send start command via Darwin notifications
  |      Main app records audio, runs FluidAudio ASR
  |      Transcription sent back via shared container + Darwin notification
  |      Keyboard extension inserts text via textDocumentProxy
  |
  +--> Cold: Open main Dicticus app via URL scheme
         Main app activates, records, transcribes
         User returns to previous app (keyboard extension resumes)
         Text inserted via textDocumentProxy
```

**Advantages:**
- Text at cursor in any app -- no clipboard, no paste
- No return-to-app problem (user stays in their app)
- On-device processing (FluidAudio) means no Full Access needed for network
- Matches what KeyVox, Wispr Flow, DictaFlow all do

**Challenges:**
- Microphone access requires main app running in background or brief app open
- IPC complexity (Darwin notifications + App Groups shared container)
- Memory constraints in keyboard extension (limited to ~55 MB)
- ASR model cannot run in the extension -- must run in main app process

### Secondary: Action Button Shortcut (Fallback)

For apps that block custom keyboards (banking, corporate, password managers):

```
Action Button press -> Dicticus app opens -> auto-records
  -> auto-stops on silence -> copies to clipboard
  -> shows "Copied! Swipe back to paste" banner
  -> user swipes back, pastes
```

This is essentially what Dicticus phase 17.5 currently implements. It should remain as a fallback for the ~5% of apps that block custom keyboards.

## Innovation Opportunities

Based on gaps observed across competitors:

1. **Haptic feedback on transcription ready:** When the Action Button shortcut finishes, trigger a haptic pulse so the user knows to swipe back and paste -- even if their phone is in their pocket.

2. **Smart clipboard with context:** When copying to clipboard, include a notification/banner that auto-dismisses, saying "Dictation copied. Paste anywhere." This is what Eloquent does and users appreciate it.

3. **Pipeline editor (Dictly-inspired):** Live text transformations during dictation -- "formal", "bullet points", "email" -- applied before insertion. Dicticus already has LLM cleanup; surfacing transformation options in the keyboard UI would differentiate.

4. **Warm session keep-alive:** KeyVox's "warm session" pattern is the key to seamless keyboard dictation. Investing in reliable background session management (audio session tricks, background task APIs) would reduce cold-start friction.

5. **Guided swipe-back (Wispr Flow pattern):** When the main app must open for cold-start mic access, show a clear visual guide: "Swipe right to return to [app name]" with an animated indicator. Flow invested heavily in making this transition less confusing.
