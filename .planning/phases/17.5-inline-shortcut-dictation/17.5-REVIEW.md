# Phase 17.5: Code Review Report

**Reviewed:** 2026-04-23T14:30:00Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

The Darwin IPC architecture for keyboard extension dictation is well-structured overall. The notification bridge pattern (Darwin notifications as signals, UserDefaults as data plane) is sound and follows established iOS patterns. The stale state reconciliation in `DicticusKeyboardIPCManager` is thoughtful. However, there are two critical issues: (1) the cancel recording handler in `DicticusApp` silently sets state to idle without actually stopping recording or cleaning up audio resources, and (2) `DicticusKeyboardIPCManager` lacks `@MainActor` isolation but its callbacks are invoked on the main queue, creating a potential data race when its properties are accessed from non-main contexts. There are also several warnings around race conditions in the heartbeat rate-limiter, missing IPC state cleanup on cancel from the host app, and a dual IPC system (legacy `kbSource`/`kbResult` alongside the new Darwin bridge) that risks desynchronization.

## Critical Issues

### CR-01: Cancel Recording Does Not Stop Audio or Clean Up Resources

**File:** `iOS/Dicticus/DicticusApp.swift:89-92`
**Issue:** The `onCancelRecordingCommand` handler only sets `viewModel.state = .idle` without calling `stopDictation()` or stopping the active audio recording. When the keyboard sends a cancel command, the main app will appear idle but `IOSTranscriptionService` will continue recording with the microphone active. The Live Activity will not be ended, the audio session will remain active, and the `hostBridge` will not publish any state update to the keyboard. This can leave the system in a silently broken state where the keyboard thinks the cancel succeeded but the app is still recording.

**Fix:**
```swift
hostBridge.onCancelRecordingCommand = {
    Task { @MainActor in
        if viewModel.state == .recording {
            // Stop audio recording and release resources
            viewModel.transcriptionService?.cancelRecording()
        }
        viewModel.state = .idle
        viewModel.hostBridge?.publishNoSpeech() // Acknowledge cancel to keyboard
    }
}
```

If `IOSTranscriptionService` does not have a `cancelRecording()` method (that stops without transcribing), one should be added. At minimum, calling `stopRecordingAndTranscribe()` and discarding the result would release resources, but a dedicated cancel method that simply stops the audio session without running inference would be better.

### CR-02: DicticusKeyboardIPCManager Lacks Actor Isolation -- Data Race on Callbacks

**File:** `iOS/DicticusKeyboard/IPC/DicticusKeyboardIPCManager.swift:11`
**Issue:** `DicticusKeyboardIPCManager` is not annotated with `@MainActor` but stores mutable callback closures (`onRecordingStarted`, `onTranscribingStarted`, etc.) that are written from the main thread (during setup) and read from the Darwin callback via `DispatchQueue.main.async`. While the callback dispatch is on the main queue, the class itself can be instantiated and have its properties accessed from any thread. With Swift 6 strict concurrency checking, this is a data race. Additionally, `DicticusKeyboardDictationController` (which is `@MainActor`) creates the IPC manager as a stored property, so accessing `ipcManager.sendStartCommand()` etc. from the `@MainActor`-isolated controller crosses isolation boundaries into a non-isolated class.

**Fix:**
```swift
@MainActor
final class DicticusKeyboardIPCManager {
    // ... rest unchanged, but darwinCallback stays nonisolated
}
```

The `darwinCallback` static closure is already `nonisolated` and dispatches to `DispatchQueue.main.async`, so it will work correctly with `@MainActor` isolation on the class. The `nonisolated` annotation on the static let handles the C callback boundary.

## Warnings

### WR-01: Heartbeat Rate-Limiter Uses nonisolated(unsafe) Static Without Synchronization

**File:** `iOS/Dicticus/IPC/DicticusIPCBridge.swift:95-108`
**Issue:** `lastHeartbeatUpdateTime` is declared `nonisolated(unsafe) private static var` and is read/written in `touchHeartbeat()` without any synchronization. `touchHeartbeat()` is called from the host bridge's Timer callback (via `Task { @MainActor in ... }`) and could theoretically be called from the keyboard extension process as well (since `DicticusIPCBridge` is compiled into both targets). While in practice this is likely only called from the main thread within a single process, the `nonisolated(unsafe)` annotation explicitly opts out of safety. If this is ever called from a background thread or concurrently, it is a data race.

**Fix:**
Either mark the entire `touchHeartbeat()` method as `@MainActor` (matching its actual call site), or use an `OSAllocatedUnfairLock` to guard the mutable state:

```swift
private static let heartbeatLock = OSAllocatedUnfairLock(initialState: TimeInterval(0))

static func touchHeartbeat() {
    let now = Date().timeIntervalSince1970
    heartbeatLock.withLock { lastUpdate in
        guard now - lastUpdate >= 1.0 else { return }
        lastUpdate = now
        defaults?.set(now, forKey: Key.sessionTimestamp)
    }
}
```

### WR-02: Dual IPC Systems Risk Desynchronization

**File:** `iOS/Dicticus/DictationViewModel.swift:102-158` and `iOS/Dicticus/DicticusApp.swift:28-41`
**Issue:** The codebase maintains two parallel IPC mechanisms for keyboard-to-app communication:
1. **Legacy:** `kbSource`, `kbResult`, `kbResultReady` UserDefaults keys (used in `DicticusApp.onOpenURL` for `dicticus://dictate` and in `DictationViewModel.stopDictation`)
2. **New Darwin bridge:** `DicticusIPCBridge` keys (`recordingState`, `latestTranscription`) plus Darwin notifications (used by `DicticusHostBridge`/`DicticusKeyboardIPCManager`)

The `dicticus://dictate` URL handler in `DicticusApp` (line 28-35) sets the legacy `kbSource` flag, while the `dicticus://record/start` URL handler (line 37-41) does NOT set `kbSource`. This means:
- Keyboard cold-start via Darwin IPC (`dicticus://record/start`) will NOT write transcription to `kbResult` (line 122-124 in DictationViewModel is gated on `isKeyboardSource`)
- The new Darwin IPC path correctly delivers results via `publishTranscriptionReady`, so this is not a functional break for the new path
- But the legacy path remains active and could be triggered, causing `kbResultReady` to be set without the Darwin `transcriptionReady` notification

**Fix:** If the legacy `kbSource`/`kbResult` path is superseded by the Darwin bridge, remove it to avoid confusion. If both paths need to coexist during migration, document clearly which URL scheme uses which IPC mechanism, and ensure the keyboard extension never mixes the two. At minimum, add a comment explaining the migration plan.

### WR-03: publishNoSpeech Double-Clears State

**File:** `iOS/Dicticus/IPC/DicticusHostBridge.swift:59-63`
**Issue:** `publishNoSpeech()` calls `DicticusIPCBridge.clearTransientOperationState()` which internally calls `setRecordingState(.idle)`, then immediately calls `DicticusIPCBridge.setRecordingState(.idle)` again on the next line. This is a minor redundancy, but more importantly, `clearTransientOperationState()` removes the `latestTranscription` key -- which means if `publishNoSpeech()` is ever called after `publishTranscriptionReady()` (e.g., in rapid succession due to a race), it will wipe the transcription text that the keyboard hasn't read yet.

**Fix:**
```swift
func publishNoSpeech() {
    DicticusIPCBridge.setRecordingState(.idle)
    DicticusIPCBridge.postDarwinNotification(named: DicticusIPCBridge.Notification.noSpeech)
}
```

Only clear `latestTranscription` if a cancel is explicitly requested, not on every "no speech" signal.

### WR-04: Cold-Start URL Doesn't Set kbSource But Legacy Cleanup Still Runs

**File:** `iOS/Dicticus/DicticusApp.swift:37-41` and `iOS/Dicticus/DictationViewModel.swift:28-35`
**Issue:** When the keyboard extension triggers a cold start via `dicticus://record/start`, `kbSource` is never set to `true`. However, `cleanupInconsistentState()` in `DictationViewModel.init()` checks for `kbSource == true` and clears it. This is not a bug per se, but on a separate cold-start scenario where the app was previously killed mid-dictation via the legacy path, `cleanupInconsistentState()` will set `kbResultReady = true` which could confuse a keyboard extension that is now using the Darwin IPC path and not polling `kbResultReady`.

**Fix:** Either remove the legacy cleanup code if the legacy IPC path is deprecated, or guard it with a version check / migration flag so it only runs when the keyboard is actually using the legacy protocol.

### WR-05: DicticusHostBridge Heartbeat Timer Closure References self Only to Prevent Deallocation

**File:** `iOS/Dicticus/IPC/DicticusHostBridge.swift:69-74`
**Issue:** The heartbeat timer closure captures `[weak self]` but then immediately does `_ = self` with a comment "prevent premature deallocation." This is contradictory -- if the intent is to prevent deallocation, the capture should be `[self]` (strong). If the intent is to allow deallocation, the `_ = self` line does nothing useful because `weak self` is already an optional that doesn't prevent deallocation. The current code has no functional bug (the Timer itself holds a strong reference to the closure, and the closure holds a weak reference to self), but the `_ = self` pattern is misleading -- it does not extend the lifetime of `self` because `self` is `Optional<DicticusHostBridge>` (weak) at that point.

**Fix:** Remove the misleading comment and the `_ = self` line. The timer will be invalidated in `deinit` or `unregisterObservers()`, which is the correct lifecycle management:

```swift
private func startHeartbeatTimer() {
    heartbeatTimer?.invalidate()
    heartbeatTimer = Timer.scheduledTimer(withTimeInterval: 1, repeats: true) { _ in
        Task { @MainActor in
            DicticusIPCBridge.touchHeartbeat()
        }
    }
}
```

## Info

### IN-01: Hardcoded App Group ID Duplicated Across Files

**File:** `iOS/Dicticus/IPC/DicticusIPCBridge.swift:15` and `iOS/Dicticus/DictationViewModel.swift:29`, `iOS/Dicticus/DicticusApp.swift:29`, `iOS/Dicticus/Intents/DictateIntent.swift:17`
**Issue:** The app group identifier `"group.com.dicticus"` is defined as a constant in `DicticusIPCBridge.appGroupID` but is hardcoded as a string literal in `DictationViewModel`, `DicticusApp`, and `DictateIntent`. These older files create `UserDefaults(suiteName: "group.com.dicticus")` directly instead of using `DicticusIPCBridge.defaults`.

**Fix:** Replace hardcoded app group strings with `DicticusIPCBridge.defaults` or `DicticusIPCBridge.appGroupID` to ensure a single source of truth:
```swift
let shared = DicticusIPCBridge.defaults
```

### IN-02: 123 Key Has No Action

**File:** `iOS/DicticusKeyboard/KeyboardExtensionView.swift:119`
**Issue:** The "123" key has an empty action closure `{ }`. Tapping it does nothing. This appears to be a placeholder for a numeric/symbol layer that hasn't been implemented yet.

**Fix:** Either implement the number row toggle or add a `// TODO: implement numeric layer` comment to clarify this is intentional.

### IN-03: Magic Numbers for Timeout Durations

**File:** `iOS/DicticusKeyboard/Dictation/DicticusKeyboardDictationController.swift:38-39` and `iOS/Dicticus/DictationViewModel.swift:220`
**Issue:** Timeout values (5 seconds, 0.5 seconds, 500ms sleep) are defined as local constants in individual files rather than being centralized. The 5-second waiting timeout in the dictation controller matches the 5-second staleness threshold in `DicticusIPCBridge.heartbeatFreshnessWindow` and the 5-second threshold in `reconcileStaleSharedStateIfNeeded`, which is good -- but this coupling is implicit, not enforced by sharing a constant.

**Fix:** Consider defining these timeout values in `DicticusIPCBridge` alongside `heartbeatFreshnessWindow` to make the relationship explicit and prevent accidental divergence.
