# Phase 17.5: Inline Shortcut Dictation — Auto-Return Flow - Research

**Researched:** 2026-04-23
**Domain:** iOS App Intents, AudioRecordingIntent, Live Activity, AVAudioSession, SpeechAnalyzer
**Confidence:** MEDIUM (core iOS constraint is HIGH confidence; auto-return UX options are MEDIUM)

---

## Summary

Phase 17.5 is an urgent insertion to deliver KEYB-02 (text at cursor without app switching) after the keyboard extension bounce approach was blocked by iOS 26 breaking all programmatic URL-opening from keyboard extensions. The goal is to make the Action Button shortcut flow fully inline: user triggers dictation, app records and transcribes, copies to clipboard, and the user returns to their previous app with minimal friction.

**The fundamental iOS constraint (HIGH confidence):** AVAudioSession microphone recording requires the app to be in the foreground. There is no supported API to record audio from a Siri Shortcut or App Intent without bringing the host app to the foreground. `openAppWhenRun = true` (or `supportedModes = [.foreground(.immediate)]`) is mandatory for any intent that triggers live audio recording. This is a platform security restriction confirmed in Apple Developer Forums (thread/756507).

**The auto-return constraint (HIGH confidence):** iOS provides no documented, supported API for an app to programmatically switch back to the previously-active app after completing an intent. This is by design. The user must manually return via swipe gesture (swipe up from bottom or use app switcher). This is confirmed by Apple DTS: "There's no supported way for you to conditionally control the behavior of opening an app like you seek with the APIs currently available." Even Wispr Flow, the leading commercial iOS dictation keyboard, shows a guidance screen and has users swipe back manually.

**Practical recommendation:** The phase goal should be re-scoped to "minimize the perceived friction of returning" rather than "automatic return." The correct approach is: app opens focused on the recording screen (no navigation, no chrome), user speaks, auto-silence-detection stops recording, transcription runs, text is copied to clipboard, Live Activity dismisses, app visually signals completion — then user swipes back. This is the same UX as Wispr Flow's "Quick Dictation to Clipboard" shortcut, which is accepted as the iOS standard for this use case.

**SpeechAnalyzer opportunity (iOS 26+ only):** iOS 26 introduces `SpeechAnalyzer` — a new on-device ASR framework with German, English, and 32+ locales, offline-first, 55% faster than Whisper Large V3 Turbo in benchmarks. It requires iOS 26+, so the current iOS 17 deployment target rules it out for now. Noted as a future migration path.

### Key Design Insight: Intent → Notification Bridge

The existing `DictateIntent.perform()` uses `pendingDictation` in shared UserDefaults and a `Notification.Name.startDictation` post. This bridge already works. The `perform()` method returns immediately (the intent is "fire and forget") — the actual recording happens in the app. This is the correct iOS pattern for `AudioRecordingIntent` with `openAppWhenRun = true`.

### Pattern 1: openAppWhenRun Required for Audio Recording

```swift
// Source: https://developer.apple.com/documentation/appintents/audiorecordingintent
// VERIFIED: Context7 + Apple Dev Forums thread/756507

struct DictateIntent: AudioRecordingIntent {
    static let title: LocalizedStringResource = "Start Dictation"
    
    // REQUIRED for microphone access — cannot be false.
    // openAppWhenRun is deprecated; use supportedModes on iOS 26+
    @available(*, deprecated)
    static let openAppWhenRun: Bool = true
    
    @MainActor
    func perform() async throws -> some IntentResult {
        // Write flag — app picks it up when it comes to foreground
        UserDefaults(suiteName: "group.com.dicticus")?.set(true, forKey: "pendingDictation")
        NotificationCenter.default.post(name: .startDictation, object: nil)
        return .result()
    }
}
```

**Why this is the only viable pattern:** AVAudioSession microphone access fails with error 561015905 when the app is not in the foreground. No workaround exists. [VERIFIED: developer.apple.com/forums/thread/756507]

### Pattern 2: Auto-Start on Shortcut Launch (App Opens Directly to Recording)

The app already handles `pendingDictation` via `checkPendingIntent()` in `DictationViewModel`. The improvement needed is:

1. App comes to foreground via intent → skip WhatsNew sheet, skip settings chrome
2. `DictationViewModel.checkPendingIntent()` fires with 500ms delay → starts recording immediately
3. No user interaction required after the app opens

The 500ms delay in `checkPendingIntent()` is intentional — it lets the AVAudioSession initialize cleanly. [ASSUMED: current value is adequate; may need tuning]

### Pattern 3: Auto-Dismiss / Minimal Chrome for Shortcut Mode

After `stopDictation()` completes, the desired UX is:

- Brief "Copied!" confirmation (1-2 seconds)
- App moves visually to a "ready to dismiss" state
- No button required to dismiss

**No API exists to programmatically minimize the app.** [VERIFIED: developer.apple.com/forums/thread/740707, Apple DTS response]

However, the Live Activity (Dynamic Island) naturally shows recording status without the user looking at the app, and dismisses automatically after 3 seconds (`dismissalPolicy: .after(.now + 3)`).

The recommended approach is a **"shortcut mode" flag** detected by `DictationView` that:
- Hides settings toolbar
- Shows only the recording UI (no history, no nav)
- Displays a "Copied — swipe back" message after completion

This is detectable via the `pendingDictation` UserDefaults key or a new `isShortcutLaunch` flag. [ASSUMED: approach, needs UX decision]

### Pattern 4: DictateIntent Can Return a String Value (QUICK-01 path)

For the future QUICK-01 requirement ("text returned as Shortcut output"), the intent can return a typed value:

```swift
// Source: https://developer.apple.com/documentation/appintents/returnsvalue
// VERIFIED: Context7 AppIntents docs

func perform() async throws -> some IntentResult & ReturnsValue<String> {
    // ... record and transcribe ...
    return .result(value: transcribedText)
}
```

This allows the Shortcut to chain: "Start Dictation → Get result → Open Notes → Paste". This is separate from Phase 17.5 scope but architecturally compatible with `AudioRecordingIntent`. [ASSUMED: QUICK-01 deferred, noting for planner]

### Anti-Patterns to Avoid

- **Trying to close the app programmatically:** No public API. `UIApplication.shared.perform(NSSelectorFromString("suspend"))` is private, App Store-rejectable.
- **Opening a URL to "return to previous app":** No universal "previous app" URL scheme exists. Per Apple DTS, this is by design.
- **Setting `openAppWhenRun = false` for audio recording:** AVAudioSession will fail with error 561015905. The Live Activity will not save it.
- **Using x-callback-url for return:** Requires the calling app to support x-callback-url (e.g., Notes does, but most apps don't). Not viable as a general solution.
- **SpeechAnalyzer on iOS 17 target:** `SpeechAnalyzer` is iOS 26+ only. Project deploys to iOS 17. Do not use.

## Common Pitfalls

### Pitfall 1: Assuming Audio Recording Can Run Without App Foreground

**What goes wrong:** Setting `openAppWhenRun = false` or using `supportedModes = [.background]` causes AVAudioSession activation to fail with error `AVAudioSessionErrorCodeCannotStartPlaying` (561015905) when triggered from Shortcuts.
**Why it happens:** iOS enforces that microphone access requires a foreground app — a security and UX policy.
**How to avoid:** Keep `openAppWhenRun = true`. The app must open. The goal is to minimize how intrusive that opening feels, not to eliminate it.
**Warning signs:** "Microphone Recording Fails When Launching from Shortcut" error in logs.
[VERIFIED: developer.apple.com/forums/thread/756507]

### Pitfall 2: checkPendingIntent Race Condition

**What goes wrong:** `checkPendingIntent()` fires before `transcriptionService` is injected, so dictation never starts despite the flag being set.
**Why it happens:** The service injection happens in `DicticusApp.onChange(of: warmupService.isReady)`, which is asynchronous. On a cold launch via intent, the service may not be ready when `setupNotificationObserver()` calls `checkPendingIntent()`.
**How to avoid:** The existing code already handles this with a dual check: once in `setupNotificationObserver()` and once in `transcriptionService.didSet`. Keep both. The 500ms delay provides additional buffer.
**Warning signs:** App opens via shortcut but stays in "idle" state without recording.

### Pitfall 3: WhatsNew / Onboarding Sheet Blocking Auto-Start

**What goes wrong:** The `WhatsNewView` sheet is presented in `DicticusApp.body` on first access after `hasSeenWhatsNewV2 = false`. If presented on top of the `DictationView`, it blocks the recording UI and the auto-start notification lands on the wrong responder.
**How to avoid:** Check `pendingDictation` flag in the `onAppear` that shows the WhatsNew sheet — suppress it if a pending dictation is in progress. Or ensure WhatsNew is marked as seen before shortcut flow can land.
**Warning signs:** App opens to WhatsNew sheet instead of recording UI.

### Pitfall 4: Live Activity Must Start Before AVAudioSession

**What goes wrong:** If `startLiveActivity()` fails (e.g., user has disabled Live Activities), the audio stops immediately per `AudioRecordingIntent` contract.
**Why it happens:** iOS enforces that `AudioRecordingIntent` must have an active Live Activity or the recording is terminated.
**How to avoid:** The existing code starts the Live Activity before `startRecording()` in `DictationViewModel.startDictation()` (Steps 2 before 3). Keep this order.
**Warning signs:** Recording starts then stops immediately with no error shown.
[VERIFIED: Context7 AppIntents docs — AudioRecordingIntent overview]

### Pitfall 5: SpeechAnalyzer Misuse

**What goes wrong:** A developer sees `SpeechAnalyzer` (iOS 26 SpeechKit) as a way to eliminate the FluidAudio dependency and do inline transcription from an App Intent.
**Why it's wrong:** `SpeechAnalyzer` requires iOS 26+. The project targets iOS 17+. Using it would exclude all iOS 17-25 users. Additionally, `SpeechAnalyzer` does not enable audio recording from a background App Intent — the same foreground requirement applies.
**How to avoid:** Do not introduce SpeechAnalyzer in this phase. Note it as a future migration path for iOS 26+ builds.

```swift
// Pattern: read shortcutLaunch flag in checkPendingIntent()
// Source: [ASSUMED] — pattern consistent with existing pendingDictation approach

private func checkPendingIntent() {
    let shared = UserDefaults(suiteName: "group.com.dicticus")
    if shared?.bool(forKey: "pendingDictation") == true {
        shared?.set(false, forKey: "pendingDictation")
        isShortcutLaunch = shared?.bool(forKey: "isShortcutLaunch") ?? false
        shared?.set(false, forKey: "isShortcutLaunch")
        Task {
            try? await Task.sleep(nanoseconds: 500_000_000)
            await self.startDictation()
        }
    }
}
```

## Open Questions

1. **Does KEYB-02 require zero app-switching, or is "minimal app-switching" acceptable?**
   - What we know: True zero-switch dictation requires microphone in keyboard extension (blocked by iOS) or background App Intent recording (blocked by AVAudioSession). Neither is possible.
   - What's unclear: Whether the user / product owner considers "app briefly opens, records, copies, user swipes back" as satisfying KEYB-02 or if it should be marked as "partially satisfied."
   - Recommendation: Confirm with user before planning. If partial satisfaction is acceptable, proceed. If zero-switch is required, this phase cannot fully deliver KEYB-02 and should say so in the plan.

2. **Should this phase also add a "Quick Dictation" App Shortcut phrase for Siri use?**
   - What we know: `DicticusShortcuts.swift` currently has "Start dictation in Dicticus" / "Dictate with Dicticus". Adding "Quick Dictation" would align with Wispr Flow's naming.
   - What's unclear: Whether Siri phrase changes require App Store submission or just app update.
   - Recommendation: Include phrase update in this phase — low cost, improves discoverability.

3. **Should the completion UI show a "Swipe to return" hint, or is this patronizing?**
   - What we know: Wispr Flow shows a guidance screen (iOS 26.4+) but experienced users find it annoying.
   - What's unclear: User preference.
   - Recommendation: Show hint only on first 3 shortcut-triggered dictations, then hide. Track with `shortcutLaunchCount` in UserDefaults.

4. **Is the iOS 17 deployment target worth raising to iOS 17.2 for `supportedModes` access?**
   - What we know: `supportedModes` was introduced in iOS 17.2. The deprecated `openAppWhenRun = true` works on iOS 17.0+.
   - What's unclear: What percentage of users are on exactly iOS 17.0 or 17.1.
   - Recommendation: Keep iOS 17.0 target. Use `@available(*, deprecated)` extension pattern for `openAppWhenRun` as already in codebase. iOS 17.0/17.1 market share is negligible (sub-2%) but backward compat costs nothing here.
