## Spacing Scale

Native SwiftUI uses point-based layout with semantic spacing constants. The 8-point grid maps to SwiftUI `.padding()` modifiers and stack spacing.

| Token | Value | SwiftUI Equivalent | Usage |
|-------|-------|-------------------|-------|
| xs | 4pt | `.padding(4)` / `spacing: 4` | Icon-to-label gaps, tight inline spacing |
| sm | 8pt | `.padding(8)` / `spacing: 8` | Row internal padding, compact section spacing |
| md | 16pt | `.padding()` (default) / `spacing: 16` | Default row padding, menu item spacing |
| lg | 24pt | `.padding(24)` / `spacing: 24` | Section separator gaps in dropdown |
| xl | 32pt | `.padding(32)` / `spacing: 32` | Onboarding panel internal margins |
| 2xl | 48pt | `.padding(48)` | Not used in Phase 1 |
| 3xl | 64pt | `.padding(64)` | Not used in Phase 1 |

Exceptions:
- Menu bar icon touch target: system-managed by macOS (NSStatusItem) — do not override
- Permission row minimum height: 44pt (HIG minimum touch target, applies to large onboarding CTA buttons)
- Onboarding panel minimum width: 360pt (fits 3-step layout without wrapping)

Source: Apple Human Interface Guidelines (macOS menu bar conventions), default applied

## Typography

All typography uses SF Pro via SwiftUI semantic text styles. Do not specify raw point sizes — use SwiftUI text style tokens so the system responds to Dynamic Type correctly.

| Role | SwiftUI Style | Approx Size | Weight | Line Height |
|------|--------------|-------------|--------|-------------|
| Body | `.body` | 13pt | Regular (400) | 1.5 (system default) |
| Label / Caption | `.caption` | 11pt | Regular (400) | 1.3 |
| Section Heading | `.headline` | 13pt | Semibold (600) | 1.2 |
| Status / Secondary | `.caption2` | 10pt | Regular (400) | 1.3 |

Rules:
- Use exactly 2 weights: Regular (400) for body/caption, Semibold (600) for headings and status labels that need emphasis
- Do not use bold (700) or heavy — menu bar UI must feel lightweight
- Status text (warm-up progress, permission state) uses `.caption` in `.secondary` color

Source: Apple HIG macOS typography, default applied; weight constraint applied from UI research

## Color

This phase uses only macOS semantic system colors. No custom hex values. System colors adapt to light/dark mode and accessibility settings automatically.

| Role | SwiftUI Semantic Token | Usage |
|------|----------------------|-------|
| Dominant (60%) | `.background` / `Color(NSColor.windowBackgroundColor)` | Menu dropdown background, onboarding panel background |
| Secondary (30%) | `Color(NSColor.controlBackgroundColor)` | Permission row backgrounds, status section backgrounds |
| Accent (10%) | `Color.accentColor` | Open System Settings buttons (CTA), active permission grant button only |
| Destructive | `Color.red` / `Color(NSColor.systemRed)` | Permission denied indicator badges only |
| Success | `Color.green` / `Color(NSColor.systemGreen)` | Permission granted indicator badges only |
| Warming | `Color.orange` / `Color(NSColor.systemOrange)` | Model warm-up in-progress indicator only |

Accent reserved for:
- "Open System Settings" button (primary CTA in permission onboarding rows)
- Active/clickable re-prompt button in degraded-state permission rows

Not used for: decorative elements, backgrounds, general text, icons other than the permission CTA

Source: Apple HIG semantic color guidance; CONTEXT.md D-02 (degraded state indicator), D-04 (warm-up indicator)

## Interaction States

### Permission Rows

Each permission row (Microphone, Accessibility, Input Monitoring) must display one of three states:

| State | Visual | Action Available |
|-------|--------|-----------------|
| Granted | Green checkmark badge + "Granted" caption | None (informational) |
| Pending / Not requested | Clock badge + "Required" caption | "Grant Access" button → opens System Settings |
| Denied | Red X badge + "Denied" caption | "Open Settings" button → opens System Settings pane |

Source: CONTEXT.md D-01 (sequential prompts), D-02 (degraded state with re-prompt)

### Model Warm-up Row

| State | Visual |
|-------|--------|
| Compiling Core ML (first launch) | ProgressView (indeterminate) + "Preparing models…" caption |
| Ready | Hidden — row disappears after warm-up completes |
| Failed | "Model load failed. Restart app." in `.caption` red text |

Source: CONTEXT.md D-03/D-04, REQUIREMENTS.md INFRA-03; multi-minute wait noted in CONTEXT.md specifics

### Dropdown Menu Structure (Phase 1 only)

```
[Menu Bar Icon]
├── [Warm-up row — hidden after ready]
│   ProgressView  "Preparing models…"
├── Divider
├── [Permission rows — always visible]
│   🟢/🔴/🕐  Microphone        [Grant Access / Open Settings]
│   🟢/🔴/🕐  Accessibility      [Grant Access / Open Settings]
│   🟢/🔴/🕐  Input Monitoring   [Grant Access / Open Settings]
├── Divider
└── Quit Dicticus
```

Source: CONTEXT.md D-05

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (permission not yet requested) | "Grant Access" |
| Primary CTA (permission denied) | "Open Settings" |
| Empty / first-launch warm-up status | "Preparing models…" |
| Warm-up complete (row hidden) | (no text — row disappears) |
| Warm-up failed | "Model load failed. Restart app." |
| Permission: granted | "Granted" |
| Permission: pending | "Required" |
| Permission: denied | "Denied" |
| Microphone row label | "Microphone" |
| Accessibility row label | "Accessibility" |
| Input Monitoring row label | "Input Monitoring" |
| Quit menu item | "Quit Dicticus" |
| Onboarding heading (first launch) | "Dicticus needs a few permissions to work" |
| Onboarding step body (microphone) | "Microphone access is required to hear your voice. Your audio never leaves this device." |
| Onboarding step body (accessibility) | "Accessibility access lets Dicticus type at your cursor in any app." |
| Onboarding step body (input monitoring) | "Input Monitoring allows Dicticus to detect your hotkey while another app is in focus." |
| Error state (permission denied inline) | "Denied — open System Settings to enable" |

Rules:
- No exclamation marks — calm and professional tone
- No jargon — "privacy" framing not "sandbox" or "entitlement"
- Keep all inline menu copy under 32 characters
- Onboarding body copy must fit in 360pt wide panel without truncation

Source: REQUIREMENTS.md APP-02; CONTEXT.md D-01/D-02 (sequential flow, degraded state); "Accessibility" wording matches System Settings label exactly to avoid confusion

## Onboarding Flow

Sequential (not simultaneous) — one permission per step.

**Step order:** Microphone → Accessibility → Input Monitoring

**Each step:**
1. SF Symbol icon for permission type (centered, 40pt)
2. Heading: permission name
3. Body: one-sentence explanation (privacy framing)
4. Primary button: "Grant Access" → calls permission API → auto-advance on grant
5. Skip link: "I'll do this later" → advances to next step, flags as pending

**Completion:**
- All granted → close onboarding panel, app is ready
- Any denied → close panel, dropdown shows degraded state rows with "Open Settings" CTAs

Source: CONTEXT.md D-01/D-02, REQUIREMENTS.md APP-02
