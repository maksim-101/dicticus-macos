# Phase 2.1: ASR Engine Swap - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace WhisperKit + Whisper large-v3-turbo with FluidAudio + Parakeet TDT v3 as the sole ASR engine. Remove all WhisperKit code, SPM dependency, and references. Preserve the TranscriptionService public API so Phase 3 can consume it unchanged.

</domain>

<decisions>
## Implementation Decisions

### Migration strategy
- **D-01:** Full rip-and-replace ‚Äî remove WhisperKit SPM dependency from project.yml, delete all `import WhisperKit` statements, and rewrite ModelWarmupService + TranscriptionService to use FluidAudio exclusively
- **D-02:** No fallback to WhisperKit ‚Äî Parakeet TDT v3 is the only ASR engine going forward
- **D-03:** No WhisperKit remnants in the codebase ‚Äî clean removal of all references, comments citing WhisperKit-specific behavior, and WhisperKit-specific test helpers

### Service API preservation
- **D-04:** TranscriptionService must preserve its public API shape (`startRecording()`, `stopRecordingAndTranscribe() async throws -> DicticusTranscriptionResult`, `State` enum, `@Published` properties) so Phase 3 hotkey wiring works without changes
- **D-05:** DicticusTranscriptionResult struct stays unchanged ‚Äî it's already abstracted from WhisperKit internals
- **D-06:** ModelWarmupService preserves its public contract (`isWarming`, `isReady`, `error`, `warmup()`, `showWarmupRow`, `statusText`) ‚Äî only internals change from WhisperKit init to FluidAudio/CoreML init

### Model distribution
- **D-07:** Follow FluidAudio's standard model loading approach ‚Äî researcher to determine whether FluidAudio downloads from HuggingFace, expects a bundled CoreML package, or offers both options
- **D-08:** First-run experience should match current behavior: model loads/compiles in background at launch with warm-up indicator

### Audio capture
- **D-09:** Researcher to determine FluidAudio's audio capture API ‚Äî whether it provides its own recording abstraction (like WhisperKit's AudioProcessor) or expects raw audio samples via AVAudioEngine

### VAD adaptation
- **D-10:** Keep the three-layer VAD defense philosophy (minimum duration guard, energy pre-filter, engine-level VAD) but adapt implementation to FluidAudio/Parakeet APIs
- **D-11:** Minimum duration threshold stays at 0.3s (D-06 from Phase 2) ‚Äî this is ASR-engine-agnostic
- **D-12:** Researcher to determine what VAD capabilities FluidAudio/Parakeet provides natively vs. what we need to implement ourselves

### Language detection
- **D-13:** Keep auto-detection restricted to {de, en} set ‚Äî Parakeet TDT v3 supports 25 languages including both; restrict post-hoc like Phase 2

### Claude's Discretion
- FluidAudio SPM integration details and version pinning
- Internal buffer management for FluidAudio's transcription API
- CoreML compilation/warm-up specifics for Parakeet model
- Energy threshold recalibration for Parakeet (may differ from Whisper)
- Test refactoring approach (adapt existing tests vs. rewrite)

</decisions>
