# Phase 02.1: Code Review Report

**Reviewed:** 2026-04-16
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

This phase swaps the ASR engine from WhisperKit to FluidAudio (Parakeet TDT v3). The overall architecture is sound: `ModelWarmupService` handles background initialization, `TranscriptionService` owns the record-resample-VAD-transcribe pipeline, and `DicticusApp` wires them together at startup. The three-layer VAD defense against Parakeet silence hallucinations is well-conceived and correctly implemented.

No critical security or data-loss issues were found. There are five warnings covering correctness risks: a race in the timeout watchdog, a missing cancellation check that leaves the warmup guard stuck, a double-idle-reset pattern that could mask errors, an AVAudioConverter callback that always claims data is available (even after the source is exhausted), and a misleading `silenceThreshold` mismatch between service default (0.5) and VAD construction (0.75). Four informational items cover dead code, an unused error case, duplicate logic, and a missing entitlement.

## Warnings

### WR-01: Timeout watchdog reads `warmupTimeoutSeconds` off a non-`@MainActor` closure

**File:** `Dicticus/Dicticus/Services/ModelWarmupService.swift:105-111`

**Issue:** The watchdog `Task` captures `self` and reads `self?.warmupTimeoutSeconds` inside a `Task { }` closure that is NOT isolated to `@MainActor`. `ModelWarmupService` is `@MainActor`-isolated, so accessing its stored properties from a non-isolated async context is a data-race under Swift 6's strict concurrency model. The `try?` swallows any `Task.sleep` cancellation silently. If the compiler does not catch this (it may pass in practice because `warmupTimeoutSeconds` is a `let` constant, which is `Sendable`), the comment "Runs separately to avoid Swift 6 Sendable issues" suggests the author is aware of tension here.

More critically: after the sleep the watchdog calls `self.cancelWarmup()` from what may be a non-`@MainActor` context (the new `Task {}` inherits the actor context of its enclosing scope — which here IS `@MainActor` because `warmup()` is `@MainActor`, so the outer `Task { }` body is also `@MainActor`). The real risk is that `isWarming` is read inside this Task without a `guard` against the service having already completed successfully; there is a TOCTOU window where `isWarming` is checked as `true` (it hasn't been set to `false` yet by the main task) and `cancelWarmup()` fires just as the main task is writing the result to `asrManager`. However because both tasks run on `@MainActor` this is cooperative, not truly concurrent — so in practice the window exists only between `await` suspension points. The check on line 108 (`if self.isWarming`) does provide a guard, but the whole watchdog task does not cancel itself when warmup succeeds; it simply sleeps for 600 seconds on every launch even when warmup finishes in 162 ms.

**Fix:** Store a reference to the watchdog task and cancel it when warmup succeeds:
```swift
// Add: private var watchdogTask: Task<Void, Never>?

watchdogTask = Task { [weak self] in
    try? await Task.sleep(nanoseconds: (self?.warmupTimeoutSeconds ?? 600) * 1_000_000_000)
    guard let self, self.isWarming else { return }
    self.cancelWarmup()
}

// In the MainActor.run success block, add:
self?.watchdogTask?.cancel()
self?.watchdogTask = nil
```

### WR-02: `cancelWarmup()` does not reset `isWarming`, leaving the guard permanently stuck

**File:** `Dicticus/Dicticus/Services/ModelWarmupService.swift:117-120`

**Issue:** `cancelWarmup()` cancels and nils out `warmupTask`, but does NOT set `isWarming = false`. The comment on line 114 says "After cancellation, the guard in `warmup()` passes again (isWarming == false, isReady == false)." This comment is incorrect: `cancelWarmup()` alone does NOT set `isWarming = false`.

The only code that sets `isWarming = false` is inside the `catch is CancellationError` branch of `warmupTask`'s body (lines 90-93). So the sequence `cancelWarmup()` → `warmupTask?.cancel()` schedules cooperative cancellation. But `cancelWarmup()` nils `warmupTask` immediately on line 119, which means if the task checks `Task.isCancelled` between the `cancel()` call and the nil assignment, it is fine. However if `cancelWarmup()` is called externally (e.g., from a future "retry" button) and the task is already done (e.g., it failed with a non-cancellation error that set `isWarming = false` via the `catch error` branch), then `warmupTask` is nil and no state change occurs — which is acceptable.

The actual bug path: if `cancelWarmup()` is called while the task is in a blocking `await` (e.g., `AsrModels.downloadAndLoad`), the task cooperative cancellation is scheduled but the task continues until the next `try Task.checkCancellation()` on line 81. Until the CancellationError propagates and is caught, `isWarming` stays `true`. If a caller calls `warmup()` in this window, the guard at line 59 prevents re-entry (correct). But the comment claiming the guard "passes again" after `cancelWarmup()` is wrong — it only passes again AFTER the CancellationError is caught asynchronously.

More concretely: if `cancelWarmup()` were called from a UI "cancel" button, the user would see the spinner forever until the cooperative cancellation propagates (which may take seconds if the download is in progress), because there is no immediate `isWarming = false` in `cancelWarmup()`.

**Fix:** Set `isWarming = false` immediately in `cancelWarmup()`, and nil the error or set an appropriate state:
```swift
func cancelWarmup() {
    warmupTask?.cancel()
    warmupTask = nil
    isWarming = false  // immediate UI feedback; task's CancellationError handler is a no-op afterwards
}
```
Also correct the misleading doc comment on line 114.

### WR-03: Double `state = .idle` assignment before throwing can mask future error handling

**File:** `Dicticus/Dicticus/Services/TranscriptionService.swift:159-178`

**Issue:** The `defer { if state == .transcribing { state = .idle } }` on line 142 is the correct pattern for resetting state. However, the three guard-throw statements after it (lines 159-169) each manually set `state = .idle` before throwing, making the defer redundant for those paths and creating a maintenance hazard. The `defer` already handles this, but the explicit resets may confuse future maintainers into thinking the defer is not active, or lead them to add state mutations before throwing that conflict with the defer.

Specifically:
- Line 160: `state = .idle` before `throw TranscriptionError.tooShort`
- Line 168: `state = .idle` before `throw TranscriptionError.silenceOnly`
- Line 175: `state = .idle` before `throw TranscriptionError.noResult`

The defer covers all these cases since `state` is `.transcribing` at that point. The manual resets are harmless but noisy.

**Fix:** Remove the redundant `state = .idle` lines before each throw. The defer handles cleanup:
```swift
guard durationSeconds >= minimumDurationSeconds else {
    throw TranscriptionError.tooShort  // defer resets to .idle
}
// ...
guard hasVoice else {
    throw TranscriptionError.silenceOnly  // defer resets to .idle
}
// ...
guard !result.text.trimmingCharacters(in: .whitespacesAndNewlines).isEmpty else {
    throw TranscriptionError.noResult  // defer resets to .idle
}
```

### WR-04: AVAudioConverter input callback always returns `haveData` — may stall or produce trailing silence

**File:** `Dicticus/Dicticus/Services/TranscriptionService.swift:240-243`

**Issue:** The `AVAudioConverter.convert(to:error:withInputFrom:)` callback on lines 240-243 always returns `sourceBuffer` and sets `outStatus = .haveData`. The correct pattern is to return `.endOfStream` (or `.noDataNow`) after the source buffer has been consumed, to signal to the converter that input is exhausted. With a single-shot source buffer that is pre-filled, the converter may call the callback more than once if the output buffer is larger than expected, receiving stale data from `sourceBuffer` on subsequent calls. This can result in duplicated or extended audio (trailing silence or audio repetition) at the end of the resampled output.

Apple's documentation specifies that the input block must set `outStatus` to `.endOfStream` once the source data is exhausted.

**Fix:**
```swift
var didProvideData = false
converter.convert(to: outputBuffer, error: &conversionError) { inNumPackets, outStatus in
    if didProvideData {
        outStatus.pointee = .endOfStream
        return nil
    }
    didProvideData = true
    outStatus.pointee = .haveData
    return sourceBuffer
}
```

### WR-05: `silenceThreshold` default (0.5) mismatches `VadManager` construction threshold (0.75)

**File:** `Dicticus/Dicticus/Services/TranscriptionService.swift:57` and `Dicticus/Dicticus/Services/ModelWarmupService.swift:79`

**Issue:** `TranscriptionService.silenceThreshold` defaults to `0.5` (line 57, with comment "Starting at 0.5 per RESEARCH.md recommendation for push-to-talk"). `VadManager` is constructed with `VadConfig(defaultThreshold: 0.75)` (ModelWarmupService line 79, also in `makeForTesting()` line 354). These are different concepts (`silenceThreshold` is compared against frame probabilities in `TranscriptionService`; `defaultThreshold` configures the `VadManager`'s internal decision boundary), but because they both affect voice detection behavior, their values should be intentionally chosen together. The current mismatch may produce unexpected results: the VadManager may internally classify frames as silence (threshold 0.75) while the TranscriptionService would pass them as voice (comparing raw probability > 0.5). A frame with probability 0.6 would pass the TranscriptionService check but may or may not be flagged by VadManager depending on how VadConfig.defaultThreshold is applied internally.

If these thresholds serve different roles (VadConfig internal vs. TranscriptionService post-filter), that should be documented explicitly. If they should be coordinated, they should share a constant.

**Fix:** Either document the intentional difference explicitly or share a constant:
```swift
// In a shared constants file or TranscriptionService:
private static let vadProbabilityThreshold: Float = 0.75

// Use when creating VadManager:
let vad = try await VadManager(config: VadConfig(defaultThreshold: Double(vadProbabilityThreshold)))

// Use in TranscriptionService:
var silenceThreshold: Float = TranscriptionService.vadProbabilityThreshold
```

### IN-01: `TranscriptionError.modelNotReady` is declared but never thrown

**File:** `Dicticus/Dicticus/Services/TranscriptionService.swift:15`

**Issue:** `TranscriptionError.modelNotReady` is defined in the enum but is never thrown anywhere in `TranscriptionService`. Because `TranscriptionService.init` requires non-optional `AsrManager` and `VadManager` parameters, a "model not ready" scenario cannot occur at the service level — it is enforced by the caller in `DicticusApp` which only creates `TranscriptionService` after warmup completes. The error case is dead code and will be confusing to future maintainers.

**Fix:** Either remove `modelNotReady` from the enum (it is enforced by the type system via non-optional init params), or add a comment explaining it is reserved for future use (e.g., Phase 3 hotkey path that may call before warmup completes).

### IN-02: `restrictLanguage(_:)` duplicates logic already in `detectLanguage(_:)` and its `#if DEBUG` static twin

**File:** `Dicticus/Dicticus/Services/TranscriptionService.swift:301-304` and `316-318`

**Issue:** `restrictLanguage` (lines 301-304) and its `#if DEBUG` static twin `testRestrictLanguage` (lines 316-318) are standalone functions that replicate the "allowed set" logic already implicit in `detectLanguage`'s `switch` statement (which only returns "de" or "en"). The `detectLanguage` method already constrains output to `{de, en}` via `languageConstraints` and its `switch`/`default: return "en"`. `restrictLanguage` is never called from within `TranscriptionService` itself — the pipeline calls `detectLanguage` directly (line 181) and does not pass the result through `restrictLanguage`. The function exists but is a no-op in the current pipeline.

**Fix:** If `restrictLanguage` is intended for callers to sanitize external input, document that and add a call in the pipeline or remove the duplicate. If `detectLanguage` already guarantees `{de, en}` output, remove `restrictLanguage` and its test wrapper.

### IN-03: Sandbox disabled in entitlements — worth a comment explaining the deliberate choice

**File:** `Dicticus/project.yml:36`

**Issue:** `com.apple.security.app-sandbox: false` disables the App Sandbox. This is intentional for a system-wide dictation app (required for Accessibility API and CGEvent injection in Phase 3), but it is a notable security tradeoff that merits documentation at the point of declaration. Without a comment, future contributors may not understand why sandboxing was explicitly disabled.

**Fix:** Add a comment in `project.yml` or the entitlements file:
```yaml
# App Sandbox disabled: required for CGEvent-based text injection into other apps (Phase 3).
# CGEvent injection via kCGEventSourceStateHIDSystemState requires non-sandboxed execution.
com.apple.security.app-sandbox: false
```

### IN-04: `testWarmupGuardPreventsDuplicateCalls` test does not actually test the guard behavior

**File:** `Dicticus/DicticusTests/ModelWarmupServiceTests.swift:96-106`

**Issue:** The test sets `isWarming = true` and then asserts that `wasWarming` (captured before the guard is exercised) is still `true`. It never calls `warmup()` to verify the guard fires. The assertion `XCTAssertTrue(wasWarming, ...)` trivially passes because `wasWarming` is set to `service.isWarming` which was just set to `true` — it is not testing the guard at all.

**Fix:** Call `warmup()` while `isWarming` is true, then check that state did not change:
```swift
func testWarmupGuardPreventsDuplicateCalls() {
    let service = ModelWarmupService()
    service.isWarming = true
    let errorBefore = service.error
    // Calling warmup() while already warming should be a no-op
    service.warmup()
    // State must remain unchanged (guard prevents re-entry)
    XCTAssertTrue(service.isWarming, "isWarming should still be true — guard prevents re-entry")
    XCTAssertEqual(service.error, errorBefore, "error should not change when guard blocks warmup()")
}
```
Note: `warmup()` starts a Task.detached — in `@MainActor` test context the task won't execute until the test yields, so state mutation from the task body will not occur synchronously, making this a valid synchronous guard test.
