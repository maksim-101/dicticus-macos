## Fixed Issues

### WR-01: Timeout watchdog not cancelled when warmup succeeds

**Files modified:** `Dicticus/Dicticus/Services/ModelWarmupService.swift`
**Commit:** ec26f99
**Applied fix:** Added a `watchdogTask` stored property to hold a reference to the timeout watchdog Task. The watchdog is now stored on creation and cancelled in all three completion paths of the warmup task: success, CancellationError, and general error. This prevents a 600-second sleeping Task from lingering after fast warm loads (~162 ms).

### WR-02: cancelWarmup() doesn't reset isWarming

**Files modified:** `Dicticus/Dicticus/Services/ModelWarmupService.swift`
**Commit:** 074d931
**Applied fix:** Added `isWarming = false` to `cancelWarmup()` for immediate UI feedback when cancellation is triggered (e.g., from a future retry button or the timeout watchdog). Updated the doc comment to accurately describe the behavior -- the guard in `warmup()` passes again immediately after `cancelWarmup()`, not only after cooperative cancellation propagates.

### WR-03: Double state = .idle assignment before throwing

**Files modified:** `Dicticus/Dicticus/Services/TranscriptionService.swift`
**Commit:** da42b30
**Applied fix:** Removed three redundant `state = .idle` assignments before `throw` statements in `stopRecordingAndTranscribe()`. The `defer { if state == .transcribing { state = .idle } }` block on line 142 already handles state cleanup for all throw paths. Added inline comments (`// defer resets to .idle`) to each throw to make the pattern explicit for future maintainers.

### WR-04: AVAudioConverter callback always returns haveData

**Files modified:** `Dicticus/Dicticus/Services/TranscriptionService.swift`
**Commit:** b0c04ea
**Applied fix:** Added a `didProvideData` flag to the `AVAudioConverter.convert(to:error:withInputFrom:)` callback. The first call returns `.haveData` with the source buffer; subsequent calls return `.endOfStream` with `nil`, per Apple's documentation. This prevents the converter from receiving stale data on repeated callback invocations, which could cause duplicated audio or trailing silence in the resampled output.

### WR-05: silenceThreshold default mismatches VadManager construction threshold

**Files modified:** `Dicticus/Dicticus/Services/TranscriptionService.swift`, `Dicticus/Dicticus/Services/ModelWarmupService.swift`
**Commit:** 9900d36
**Applied fix:** Introduced `TranscriptionService.vadProbabilityThreshold` (0.75) as a shared static constant. Updated three sites to use it: (1) `silenceThreshold` default in TranscriptionService, (2) `VadConfig(defaultThreshold:)` in ModelWarmupService.warmup(), (3) `VadConfig(defaultThreshold:)` in the `#if DEBUG` `makeForTesting()` helper. Added documentation explaining why both VadManager and TranscriptionService share the same threshold value.
