# Phase 20.06 Plan 03: iOS History Row UX Hotfix Summary

Added a `.contextMenu` Copy action and a trailing `chevron.right` affordance to `HistoryRow` on iOS, fixing two UAT-2026-04-27 findings (F-20-UAT-03 long-press popping a system path/link preview; F-20-UAT-04 row-tap navigation being undiscoverable). One file modified, two atomic commits, no Shared/ or NavigationLink changes.

## What Changed

### iOS/Dicticus/History/HistoryView.swift

The `HistoryRow` body root changed from a single VStack to an `HStack(alignment: .center, spacing: 8)` containing:

1. The original VStack content (date+lang HStack, `Text(entry.text).lineLimit(3)`, confidence+inline-Copy HStack) wrapped with `.frame(maxWidth: .infinity, alignment: .leading)` so it occupies the leading area.
2. A trailing `Image(systemName: "chevron.right")` styled `.font(.caption.weight(.semibold)).foregroundStyle(.tertiary)` and marked `.accessibilityHidden(true)`.

The modifier chain on the row root (the new outer HStack) is:

```swift
.padding(.vertical, 4)
.contextMenu {
    Button {
        copyToClipboard()
    } label: {
        Label("Copy", systemImage: "doc.on.doc")
    }
}
.accessibilityElement(children: .combine)
```

Order matters: `.contextMenu` is placed between `.padding(.vertical, 4)` and `.accessibilityElement(children: .combine)` so the long-press gesture is registered at the row-root level (overriding any system Text long-press auto-detection on the inner `Text(entry.text)`), and the combined-element accessibility shape still wraps the entire row including the chevron.

The `.contextMenu` Copy button calls the existing `copyToClipboard()` method, which honors `CleanupCopyMode.current` (raw vs polished) — so the long-press Copy and the inline Copy button copy the same text in the same way.

## Final Shape (HistoryRow body)

```
HStack(alignment: .center, spacing: 8) {
    VStack(alignment: .leading, spacing: 8) {
        // date + language pill

.accessibilityElement(children: .combine)
```
